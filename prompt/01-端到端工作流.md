# 01 · 端到端工作流

> 用本文档作为"项目从 0 到编码前"的总指挥手册。每一步都对应 `/prompt` 下的一份模板。

---

## 一、阶段总线

```mermaid
flowchart LR
  A[R1 需求初稿] --> B[R2 AI 澄清提问]
  B --> C[R3 需求分析报告]
  C -->|R 冻结| D[P1 原型方向]
  D --> E[P2 AI 澄清提问]
  E --> F[P3 信息架构 + 页面清单]
  F --> G[P4 HTML 原型]
  G -->|迭代到稳定| H[P 冻结]
  H --> I[G1/G2/G3 全局规范]
  I -->|G 冻结| J[F1 数据]
  J --> K[F2 接口]
  K --> L[F3 页面交互]
  L -->|F 冻结/按功能| M[V1 一致性校验]
  M --> N[V2 Story 拆分]
  N --> O[V3 编码上下文打包]
  O --> P[进入编码]
```

---

## 二、阶段闸门（Gate）

每一阶段必须通过下表中的 Gate 才能进入下一阶段。Gate 不通过 → 回到本阶段重做。

| Gate | 名称 | 检查项 | 通过判定 |
|------|------|--------|----------|
| G-R | 需求闸门 | R3 报告中无 `[待确认]`、流程图覆盖所有主路径与异常路径、用户已签字 | 所有需求条目都有唯一 ID 且"是否冻结"=Y |
| G-P | 原型闸门 | P4 HTML 原型在浏览器中可点通，所有页面跳转可达，关键状态（空/错/loading/无权限）都有静态页 | PM 三轮以内不再提改动 |
| G-G | 全局闸门 | G1/G2/G3 三份 AI 输出无 `99-待确认问题`，编码风格、Token、权限角色全部锁死 | 三份 AI 输出文件签字归档 |
| G-F | 功能闸门 | 单功能的 F1+F2+F3 互相引用一致：F2 中每个接口的入参/响应字段都能在 F1 找到；F3 中每个数据字段都能在 F1 找到；F3 中每个 API 调用都能在 F2 找到 | V1 一致性报告通过 |
| G-V | 校验闸门 | V1 报告无红色项；V2 Story 颗粒度均匀（每个 Story ≤ 半天）；V3 编码上下文包尺寸 ≤ 1200 行/包 | 三份 V 文件签字 |

> Gate 不通过严禁强行进入下一阶段——这是本框架抗腐化的唯一防线。

---

## 三、文件流（按顺序产出）

```
content/<project>/requirements/
  baseline.md                ← R3 产物（冻结后）
  questions-resolved.md      ← R2 + 用户回答合并
content/<project>/prototype/
  ia.md                      ← P3 信息架构
  pages.md                   ← P3 页面清单
  html/                      ← P4 HTML 原型目录
  changelog.md               ← 每轮原型迭代记录
grules/<project>/
  G1-架构与技术规范/...        ← G1 AI 输出（沿用 /grules 现有结构）
  G2-视觉与交互风格/...
  G3-权限与认证规范/...
function/<project>/<feature>/
  F1-数据模型规范.md           ← F1 AI 输出
  F2-接口规范.md
  F3-页面交互规范.md
  _v/
    V1-一致性校验.md
    V2-story.md
    V3-coding-context-bundle.md
```

## 四、AI 对话上下文喂养策略（最小必要原则）

每一步开新对话时，**只**把下表列出的文件作为系统消息上传，禁止"以防万一全塞进去"。

| 步骤 | 系统消息（上文） | 用户消息（本轮输入） |
|------|----------------|-------------------|
| R2 提问 | 00, 02, 10, 12 | R1 用户填好的需求初稿 |
| R3 报告 | 00, 02, 10, 13, R2 提问 + 用户回答合并版 | "按 13 模板出报告" |
| P2 提问 | 00, 02, 20, 22, R3 基线 | P1 用户填好的原型方向 |
| P3 IA  | 00, 02, 20, 23, R3 基线, P2 已答 | "按 23 模板出 IA" |
| P4 HTML | 00, 02, 20, 24, P3 产物 | "按 24 规范出 HTML 原型，每页一个文件" |
| G1 提问 | 00, 02, 30, /template/all_totals/G1-AI | G1 用户输入 |
| G1 输出 | 00, 02, 30, /template/all_totals/G1-AI, G1 提问已答 | "按模板出 G1 规范" |
| G2/G3 | 同上规则替换 | … |
| F1 提问 | 00, 02, 40, /template/all_totals/F1-AI, G1 输出 | F1 用户输入 |
| F1 输出 | 同上 + F1 提问已答 | "按模板出 F1" |
| F2/F3 | 按 40 文件规则补充上游 | … |
| V1 校验 | 00, 02, 50, 当前功能的 F1+F2+F3+G1+G2+G3 | "按 50 出 V1 报告" |
| V2 Story | 00, 02, 50, V1 通过版 | "按 50 V2 部分拆 Story" |
| V3 打包 | 00, 02, 50, V2 拆完 | "按 50 V3 部分打编码上下文包" |

> 反例：把整个 `/grules` 全塞进 R 阶段——R 阶段是分析师，不该看技术细节。这会污染分析。

## 六、新项目启动 Checklist

- [ ] 在 `content/<project>/`、`function/<project>/`、`grules/<project>/` 创建空目录
- [ ] 复制 `11-R1` 模板，填业务诉求
- [ ] 完成 R 阶段三件套，过 Gate G-R
- [ ] 复制 `21-P1` 模板，填原型方向
- [ ] 完成 P 阶段三件套，迭代到稳定，过 Gate G-P
- [ ] 复制 `/template/all_totals/G1/G2/G3-用户` 模板，按 30 文件编排做完三份全局规范，过 Gate G-G
- [ ] 列出本期所有功能，逐功能跑 F 阶段三件套，过 Gate G-F
- [ ] 跑 V 阶段三件套，过 Gate G-V
- [ ] 进入编码

> 任何一步偷懒跳过 → 后果由偷懒人承担。框架不背锅。
