# A00-01 · 框架总览

## 一、定位

一套从「业务需求」到「开发规范」全链路的 AI 协同提示词模板框架。按 **F（定调）/ C（产品设计）/ D（开发规范）** 三层组织，共 **7 阶段**。

每阶段 AI 严格按模板产出文档，所有产物以可被下游机械消费为目标。

---

## 二、三层结构与 7 阶段速查

### F 层 · 一次性定调（项目级，跑一次）

| 序号 | 阶段 | 角色 | 步骤 | 输出概要 |
|------|------|------|------|---------|
| 1 | B01·A 技术架构 | 架构师 | 2 步（用户输入 → AI 输出） | 选栈、目录、DB/API/编码规范、surface 清单、鉴权基础设施 |
| 2 | B02·XS 体验与设计系统 | 体验总监 + UI 设计师 | 2 步（用户输入 → AI 输出） | 产品调性 + 设计 Token、组件、布局、原型样式包 |

### C 层 · 按功能循环（每个功能跑一遍）

| 序号 | 阶段 | 角色 | 步骤 | 输出概要 |
|------|------|------|------|---------|
| 3 | C01·RP 需求与权限 | 需求分析师 + 权限工程师 | 2 步（用户输入 → AI 输出） | 需求清单 R-ID、用户故事、验收标准、角色权限矩阵 |
| 4 | C02·IN 架构与交互 | 信息架构师 + 交互设计师 | 2 步（用户输入 → AI 输出） | 功能清单、流程图、状态机、页面清单、单页布局与行为 |
| 5 | C03·H HTML 原型 | 原型工程师 | 1 步（AI 直接输出） | 可交互跳转的连通 HTML 原型，用 `python -m http.server` 运行 |

### D 层 · 按功能循环（C 层完成后）

| 序号 | 阶段 | 角色 | 步骤 | 输出概要 |
|------|------|------|------|---------|
| 6 | D01·D 数据规范 | 数据建模师 | 1 步（AI 直接输出） | 表结构、枚举、字段校验、索引 |
| 7 | D02·L 接口规范 | 接口设计师 | 1 步（AI 直接输出） | 全局路由表、接口契约、错误码、并发策略 |

---

## 三、走序

```
[一次性] B01·A → B02·XS → F 层全部 ✅

[按功能循环] for each <功能>:
  C01·RP → C02·IN → C03·H → D01·D → D02·L → 进入该功能开发
```

- F 层全部冻结后才能开始任何功能的 C 循环
- C03·H 原型稳定后才能启动 D 层（设计先行）
- 每轮循环围绕**一个功能**（模块内子单元），粒度示例：`account-entry`（属 `auth` 模块）、`learning`（属 `course` 模块）；模块（如 `auth`、`course`）是多个功能的集合，不是循环粒度

---

## 四、核心原则

| 原则 | 说明 |
|------|------|
| 设计先行 | 先用原型钉死产品形态，再倒推数据与接口 |
| 功能粒度循环 | C/D 层以功能为单位循环，每轮产出一个功能的完整设计 |
| 增量融合 | 每轮新功能的产出增量融合到已有的全局文档中（权限、路由表等） |
| 上下文最小化 | 每阶段 AI 只看当下需要的上游文件，禁止全塞 |
| 角色独立窗口 | 不同阶段使用独立对话窗口，互不污染 |

---

## 五、阶段角色边界

| 阶段 | 严禁做的事 |
|------|-----------|
| B01·A 架构师 | 不写功能字段、不写页面、不写业务角色枚举 |
| B02·XS 体验+设计 | 不定接口、不定页面交互逻辑 |
| C01·RP 需求+权限 | 不画详细流程图、不定状态机、不出 mermaid、不做技术选型 |
| C02·IN 架构+交互 | 不定接口、不写代码、不出真实表名/SQL |
| C03·H 原型 | 不动业务规范、不写后端代码 |
| D01·D 数据 | 不定路由/接口、不定页面 |
| D02·L 接口 | 不写 SQL、不写 UI |

## 六、步骤说明

**2 步阶段**（B01·A、B02·XS、C01·RP、C02·IN）：
1. 用户填写输入模板
2. AI 直接产出规范文档

**1 步阶段**（C03·H、D01·D、D02·L）：
1. AI 基于上游已冻结产物直接输出，无需用户新增输入

## 七、文件命名规则

- 框架层：`A00-<序号>-<名称>.md`
- 阶段模板：`<阶段码>-<步骤号>-<类型>-<名称>.md`
  - 类型：`用户输入` / `AI输出`
  - 示例：`B01-01-用户输入-技术偏好.md`、`C01-02-AI输出-需求与权限基线.md`
- 模块 ID：全小写短横分隔，顶层业务域（`auth`、`course`、`order`）
  - 功能 ID (feature)：全小写短横分隔，模块内子功能（`account-entry`、`learning`、`checkout`）

## 八、全局输出目录结构

所有阶段的 AI 产出统一落盘到项目根目录下。原型目录与规范目录平级，确保 `python -m http.server` 可直接运行原型。

```
<project-root>/
  prompt/                              # 框架模板（本目录，只读）
  docs/                                # 所有 markdown/规范产出
    B01-architecture/                  # B01 输出（11 文件）
    B02-experience-design/             # B02 输出
      design-system/                   #   设计系统 markdown
      prototype-style/                 #   CSS/JS 可执行资产（被 C03 引用）
    C01-requirements/                  # C01 输出
      permissions/                     #   共享权限文件（按端拆分）
      <module-id>/<feature-id>/        #   功能需求基线
    C02-ia-interaction/                # C02 输出
      <module-id>/<feature-id>/        #   信息架构与交互规范
    D01-data/                          # D01 输出
      <module-id>/<feature-id>/        #   数据规范
    D02-api/                           # D02 输出
      <module-id>/<feature-id>/        #   接口规范
  C03-prototype/                       # C03 输出（与 docs/ 同级，可独立运行）
    index.html                         #   项目总览
    <module-id>/                       #   模块目录
      index.html                       #   模块功能目录页
      _shared/                         #   模块共享组件
      <feature-id>/                    #   功能目录
        <page-id>.html                 #   演示页面
```

> C03-prototype 与 docs/ 同级（非嵌套），因此原型页面引用样式包的相对路径为 `../docs/B02-experience-design/prototype-style/`（从项目总览计）。

## 九、冻结（Freeze）定义

- **含义**：文件头部 `冻结状态` 标为 `已冻结`，表示内容稳定，下游可引用。
- **前置条件**：该文件 §99 待确认问题一节为空（所有疑问已补答）。
- **冻结后修改规则**：冻结后仅可通过增量融合追加内容（标注 `[本轮新增]` / `[本轮变更]`），不得删除或重写已有冻结内容。如需重大修改，必须回退到对应阶段重新通过 Gate。
- **F 层冻结**：B01 + B02 全部冻结后方可启动任何功能的 C 循环。
- **C 层冻结**：C01 → C02 → C03 依次冻结后方可启动 D 层。

---

## 十、C 层输出结构

C 层产出按 **模块 → 功能 → 端** 二级细分组织：

```
模块（module）
  └─ 功能（feature）
       └─ 端（surface）：app / admin / ...
```

单端项目不拆 surface 子目录。多端项目在 B01 中定义 surface 清单，后续按需拆分。
