# 01 · G01 · 框架总览

> 本文件回答三个问题：本框架是什么、为什么这样设计、怎么使用。

---

## 一、是什么

一套从「业务诉求」到「上线增长」全链路的标准化提示词模板集，按 **A（框架）/ B（项目定调）/ C（产品设计循环）/ D（开发循环）** 四个文件夹组织，共 4 + 12 + 15 + 9 份模板。

它不替你写代码，它让 AI 在每一步都按你定义的格式做事，并在所有不清楚的地方主动问你，直到产出可以被下游机械消费的文档。

---

## 二、为什么这样设计

AI 协同开发最常见的三个翻车点：

1. **AI 凭空脑补**：用户没说的细节，AI 直接默认了，下游全错。
2. **上下文塞太多**：一次喂 1 万字，AI 抓不住重点，丢字段。
3. **角色串场**：让「画原型的 AI」同时定数据库字段，结果两头不到位。

本框架的三个解药：

| 解药 | 落地形态 |
|------|---------|
| 强制澄清协议 | 每一步必经「用户填表 → AI 必先提问 → 用户答 → AI 出报告」，AI 跳步即打回 |
| 上下文最小化 | 每一步明确「系统消息只塞这几份文件」，禁止全塞 |
| 角色独立窗口 | 12 阶段 × 不同角色，独立对话窗口，互不污染 |

---

## 三、四层结构（A / B / C / D）

第四个翻车点：**先建表后改 UI**——传统瀑布式把数据模型放在最前，等做完原型才发现字段不够 / 多余 / 形态错位，被迫推翻重来。

解药 4：**先用原型把产品形态钉死，再倒推数据与接口**。以文件夹表达分层，严格从上到下、从左到右：

```
prompt/
├─ A-framework/        # A 、一次性，读过即可
│   └─ A00 框架/工作流/约定/目录规划
├─ B-foundation/       # 项目级、一次性：冻结后才能开任何 feature
│   ├─ B01 A 技术架构
│   ├─ B02 P 权限与角色
│   ├─ B03 X 体验定调
│   └─ B04 S 设计系统
├─ C-product/          # feature 级产品设计循环：可独立走完交付 PRD
│   ├─ C01 R 需求分析
│   ├─ C02 I 信息架构
│   ├─ C03 N 页面交互
│   ├─ C04 H HTML 原型
│   └─ C05 E 产品需求文档
└─ D-develop/      # feature 级开发循环：PRD 冻结后才走
    ├─ D01 D 数据模型
    ├─ D02 L 业务与接口
    └─ D03 V 开发前一致性校验
```

**走序**：B 全部 ✅ → (针对某 feature) C01→C02→C03→C04→C05 → D01→D02→D03 → 进入该 feature 开发。

**多应用端（surface）**：项目可能同时存在多个端（`app` / `admin` / `merchant` / `open` …）。本框架采用**自适应复杂度**：单端项目什么都不变；多端项目在 B01 `08-surfaces.md` 里定 surface 清单，后续 C/D 阶段产出物按 `<feature>/<surface>/` 拆子目录（详 A00-04）。跨端强关联（如 admin 配商品 → app 看到上架）靠共享的 R 需求基线 + 共享的 D 数据模型 + 共享的状态机保证一致。

**「登录」不是全局事**：B02 不定任何登录/注册/忘密流程。每个 surface 有自己的 `<surface>-auth` feature（如 `app-auth`、`admin-auth`），与「商品」「订单」同类别，走常规 C01→C05 循环，共用 B01 `09-auth-infra.md` 定下的鉴权基础设施。

**C 与 D 的隔离**：C01–C05 **不允许引用 B01（架构）与 D 阶段产物**，只靠 B02/B03/B04 + 本 feature 上一步产物。这让你只拉 A+B+C 三个文件夹即可交付一份产品设计包（含可点原型 + PRD），无需踏进开发。

## 四、12 阶段执行顺序与依据

### F · 一次性定调（项目启动后跑一次）

| 顺序 | 阶段 | 为什么这个时机 |
|------|------|--------------|
| 1 | B01 · A 技术架构 | 选栈、定目录、定 DB/API/编码规范、**定 surface 清单**、定鉴权基础设施——后续所有产物都按它走 |
| 2 | B02 · P 权限与角色 | 角色枚举 + 角色×surface 矩阵要先定；不包含登录/注册/忘密流程（那些是 feature 级）|
| 3 | B03 · X 体验定调 | 决定「产品给人的感觉」；先于视觉系统 |
| 4 | B04 · S 设计系统 | 把 X 的调性落成 Token、组件、布局；同时输出可运行的原型样式包（`prototype-style/`），C04 原型直接 vendor 引用其 CSS / JS 资产 |

### C · 按 feature 循环（每个 feature 走一遍 8 步）

| 顺序 | 阶段 | 为什么这个时机 |
|------|------|--------------|
| 1 | C01 · R 需求分析 | feature 启动第一件事——把这个 feature 的目标、用户故事、验收标准、需求清单（R-ID）结构化 |
| 2 | C02 · I 信息架构 | 产品经理视角梳理功能清单、业务流程图、状态机、页面清单、导航结构；**不产出 URL / 路由**（路由是开发件，放 D02）|
| 3 | C03 · N 页面交互 | B04 设计系统 + C02 页面/功能清单到位后，精准描述每页元素与行为 |
| 4 | C04 · H HTML 原型 | 把前 3 步「活」出来：可点通、4 态齐、有真数据型态。本 feature 全部 page-id 一次性出齐 |
| 5 | C05 · E PRD（feature 级）| 原型稳定后回写产品规格，作为本 feature「任何人接手即可读懂」的总章 |
| 6 | D01 · D 数据模型 | **倒推**：从已稳定的 PRD + 原型 + C02 状态机 反推表结构 / 校验，不会建错字段 |
| 7 | D02 · L 路由与接口 | 数据有了，按页面行为定义全局路由表与所有接口：URL 规则 / endpoint / 错误码 / 并发 |
| 8 | D03 · V 一致性校验 | V01 上游链 + V02 模块闭环 + V03 PRD 回链；三颗全绿才进开发 |

> **「设计先行」核心理由**：用户对产品形态的判断在原型上最快收敛；数据 / 接口对原型的服从代价远小于反向。
> **多 feature 并行**：F 冻结后，feature A 和 feature B 可同时跑各自 C 循环，互不阻塞；只通过全局 PRD 索引 `docs/C05-prd/_global-index.md` 协调。

## 五、阶段角色

### F · 一次性定调

| 阶段 | 角色 | 单一职责 | 严禁做的事 |
|------|------|---------|-----------|
| B01 · A | 架构师 | 选栈、定规范、定目录、**定应用端清单（surface）**、定鉴权基础设施 | 不写功能字段、不写页面、不写登录页交互 |
| B02 · P | 安全/权限工程师 | 定角色枚举、角色×surface 矩阵、授权机制、审计、数据可见范围 | 不定登录/注册/忘密流程（这是 feature 级 `<surface>-auth`）、不实现具体业务接口 |
| B03 · X | 体验总监 | 产品调性、参照、品牌情绪 | 不画具体组件 |
| B04 · S | UI 设计师 | Token、组件、布局、状态色 | 不定接口、不定页面交互 |

### C · 按 feature 循环

| 阶段 | 角色 | 单一职责 | 严禁做的事 |
|------|------|---------|-----------|
| C01 · R | 需求分析师 | 把 feature 模糊业务 → 结构化需求清单（R-ID） | 不画详细流程、不定状态机、不做技术选型、不画 UI |
| C02 · I | 信息架构师 | 本 feature 的功能清单 / 业务流程图 / 状态机 / 页面清单 / 导航结构 | 不出 URL / 路由、不画单页布局、不设计接口 |
| C03 · N | 交互设计师 | 单页布局 / 元素 / 行为 / 4 态 | 不定接口、不写代码 |
| C04 · H | 原型工 | 出可点通 HTML | 不动业务规范 |
| C05 · E | 产品经理 | 基于已稳定原型 + 上游沉淀输出 feature 级 PRD | 不新增需求、不改字段、不写代码 |
| D01 · D | 数据建模师 | 表结构、枚举、字段校验、索引 | 不定路由/接口、不定页面、不重定义 C02 状态机 |
| D02 · L | 路由/接口设计师 | 全局路由表、URL 规则、接口契约、错误码、并发策略 | 不写 SQL、不写 UI |
| D03 · V | QA / 文档审计师 | V01 上游链 / V02 模块闭环 / V03 PRD 回链 | 不写业务代码、不拆 Story、不打编码包 |

## 六、阶段闸门（Gate）

每阶段必经闸门才能进入下一阶段。任何 Gate 不通过 → 回到本阶段重做，**严禁强行往下推**。

### F · 一次性闸门（4 颗，全部 ✅ 才能开任何 feature 的 C）

| Gate | 通过判定 |
|------|---------|
| G-A | 架构规范无遗留问题、技术栈/目录/DB/API/编码规范全部锁死、surface 清单与鉴权基础设施锁死 |
| G-P | 角色枚举、角色×surface 矩阵、授权机制锁死；未出现任何登录/注册页产物 |
| G-X | 体验定调有 ≥3 个参照、≥3 个反例、≥5 个可验证形容词 |
| G-S | 设计 Token 完备、所有组件都有 4 态、与 X 调性吻合 |

### C · 按 feature 闸门（8 颗，逐 feature 跑）

| Gate | 通过判定 |
|------|---------|
| G-R | feature 需求基线无 `[待确认]`、需求清单 R-ID 唯一、用户签字 |
| G-I | 本 feature 功能清单、流程图、状态机、页面清单均被 R-ID 全覆盖；状态机闭合 |
| G-N | 单页规范元素齐全、4 态齐全、引用 B03/B04 设计系统对齐 |
| G-H | 原型可点通、4 态齐全、用户三轮内不再大改 |
| G-E | feature PRD 多文件齐；每句话可回链上游 ID；术语统一；99 节为空 |
| G-D | 数据规范字段齐、校验完整、覆盖 PRD 中所有数据形态；不重定义 C02 状态机 |
| G-L | 全局路由表与 C02 页面清单一一对齐；接口规范覆盖所有页面行为与状态转移、错误码完备、与 D 字段一一对齐 |
| G-V | V01/V02/V03 三份报告皆无红色项；所有 99 节为空；取得后**才**进入本 feature 开发 |

---

## 七、最小用法

```
[一次性] 读 A-framework/A00-04 建好 docs/ 目录
[一次性] B-foundation/B01 A → B02 P → B03 X → B04 S → 全部 ✅
        【B04 双产出】docs/B04-design/design-system/（markdown 规范）
                     docs/B04-design/prototype-style/（可运行 CSS / JS 资产，C04 原型 vendor 引用）

[按 feature 循环 A] for each <feature>:
  C-product/C01 R → C02 I → C03 N → C04 H（一次性出齐）→ C05 E（多文件 PRD）
  ——到此可交付产品设计包，不必继续。

[按 feature 循环 B] 需要入开发时再走:
  D-develop/D01 D → D02 L → D03 V（V01+V02+V03 全绿）→ 进入开发
```

每一步只做一件事，AI 只看当下需要的文件。
