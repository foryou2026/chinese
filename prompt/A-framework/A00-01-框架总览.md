# 01 · A00-01 · 框架总览

## 一、定位

一套从「业务诉求」到「上线增长」全链路的标准化提示词模板集，按 **A（框架）/ B（项目定调）/ C（产品设计循环）/ D（开发循环）** 四个文件夹组织，共 4 + 9 + 18 + 9 份模板。

每一步要求 AI 严格按指定格式产出文档，未澄清处必须先提问；所有产物以可被下游机械消费为目标。

---

## 二、核心原则

| 原则 | 落地形态 |
|------|---------|
| 强制澄清协议 | 每一步必经「用户填表 → AI 必先提问 → 用户答 → AI 出报告」，AI 跳步即打回 |
| 上下文最小化 | 每一步明确「系统消息只塞这几份文件」，禁止全塞 |
| 角色独立窗口 | 12 阶段 × 不同角色，独立对话窗口，互不污染 |
| 设计先行 | 先用原型把产品形态钉死，再倒推数据与接口（D01/D02 放在 C05/C06 之后） |

---

## 三、四层结构（A / B / C / D）

以文件夹表达分层，严格从上到下、从左到右：

```
prompt/
├─ A-framework/        # A · 框架本身，一次性，读过即可
│   └─ A00 README / 总览 / 工作流 / 通用约定 / 目录规划
├─ B-foundation/       # B · 项目级、一次性：冻结后才能开任何 feature
│   ├─ B01 A 技术架构（含 surface 清单 + 鉴权基础设施）
│   ├─ B02 X 体验定调
│   └─ B03 S 设计系统 + 可运行原型样式包
├─ C-product/          # C · feature 级产品设计循环：可独立走完交付 PRD
│   ├─ C01 R 需求分析
│   ├─ C02 P 角色与权限（增量并入全局 docs/C02-permissions/）
│   ├─ C03 I 信息架构
│   ├─ C04 N 页面交互
│   ├─ C05 H HTML 原型
│   └─ C06 E 产品需求文档（PRD）
└─ D-develop/          # D · feature 级开发循环：PRD 冻结后才走
    ├─ D01 D 数据模型
    ├─ D02 L 业务与接口
    └─ D03 V 开发前一致性校验
```

**走序**：B 全部 ✅ → (针对某 feature) C01→C02→C03→C04→C05→C06 → D01→D02→D03 → 进入该 feature 开发。

**P（权限）执行位置**：放在 feature 的 C 循环，本 feature 的角色/权限增量并入全局 `docs/C02-permissions/`；F 层只负责鉴权基础设施（在 `B01-architecture/09-auth-infra.md`，仅当项目有登录需求时才有实质内容）。

**多应用端（surface）**：项目可能同时存在多个端（`app` / `admin` / `merchant` / `open` …）。单端项目什么都不变；多端项目在 B01 `08-surfaces.md` 里定 surface 清单，后续 C/D 阶段产出物按 `<feature>/<surface>/` 拆子目录（详 A00-04）。跨端强关联（如 admin 配商品 → app 看到上架）靠共享的 R 需求基线 + 共享的 D 数据模型 + 共享的状态机保证一致。

**feature 命名**：业务功能（商品、订单、内容）、登录/认证、设置等，都是普通 feature，命名都是 kebab-case 业务语义（如 `auth`、`payment`、`settings`）。如果某 feature 同时在多端出产物（如 `auth` 同时在 `app` 与 `admin`），就按多端形态 `<feature>/<surface>/` 走；只在 1 端出产物就只建 1 个 surface 子目录；项目没有该 feature（如不需登录）就不出现。

**C 与 D 的隔离**：C01–C06 **不允许引用 D 阶段产物**，只靠 B 层 + 本 feature 上一步产物。

## 四、12 阶段执行顺序

### F · 一次性定调（项目启动后跑一次，3 步）

| 顺序 | 阶段 | 输出 |
|------|------|------|
| 1 | B01 · A 技术架构 | 选栈、定目录、定 DB/API/编码规范、定 surface 清单、定鉴权基础设施（GoTrue / JWT / Cookie 策略 / 中间件签名，仅当项目有登录类需求时） |
| 2 | B02 · X 体验定调 | 决定「产品给人的感觉」 |
| 3 | B03 · S 设计系统 | 把 X 的调性落成 Token、组件、布局；同时输出可运行的原型样式包（`prototype-style/`），C05 原型直接相对路径引用其 CSS / JS 资产（不拷贝） |

### C · 按 feature 循环（每个 feature 走一遍 9 步）

| 顺序 | 阶段 | 输出 |
|------|------|------|
| 1 | C01 · R 需求分析 | 本 feature 的目标、用户故事、验收标准、需求清单（R-ID） |
| 2 | C02 · P 角色与权限 | 本 feature 引入的新角色 / 新权限粒度 / 新数据可见范围 / 新审计点，增量并入 `docs/C02-permissions/`；如未引入新内容则只产「无新增」说明 |
| 3 | C03 · I 信息架构 | 功能清单、业务流程图、状态机、页面清单、导航结构；不产出 URL / 路由（路由是开发件，放 D02）|
| 4 | C04 · N 页面交互 | 每页元素与行为 |
| 5 | C05 · H HTML 原型 | 可点通、4 态齐、有真数据型态。本 feature 全部 page-id 一次性出齐，并配套引导页（index.html）三级嵌套 |
| 6 | C06 · E PRD（feature 级）| 原型稳定后回写产品规格 |
| 7 | D01 · D 数据模型 | 从已稳定的 PRD + 原型 + C03 状态机反推表结构 / 校验 |
| 8 | D02 · L 路由与接口 | 全局路由表、URL 规则、endpoint、错误码、并发策略 |
| 9 | D03 · V 一致性校验 | V01 上游链 + V02 模块闭环 + V03 PRD 回链；三颗全绿才进开发 |

## 五、阶段角色

### F · 一次性定调

| 阶段 | 角色 | 单一职责 | 严禁做的事 |
|------|------|---------|-----------|
| B01 · A | 架构师 | 选栈、定规范、定目录、定应用端清单（surface）、定鉴权基础设施（如项目有登录类需求） | 不写功能字段、不写页面、不写业务角色枚举 |
| B02 · X | 体验总监 | 产品调性、参照、品牌情绪 | 不画具体组件 |
| B03 · S | UI 设计师 | Token、组件、布局、状态色 | 不定接口、不定页面交互 |

### C · 按 feature 循环

| 阶段 | 角色 | 单一职责 | 严禁做的事 |
|------|------|---------|-----------|
| C01 · R | 需求分析师 | 把 feature 模糊业务 → 结构化需求清单（R-ID） | 不画详细流程、不定状态机、不做技术选型、不画 UI、不出 mermaid |
| C02 · P | 安全/权限工程师 | 本 feature 新增角色、权限矩阵、数据可见范围、审计点；增量并入全局 `docs/C02-permissions/` | 不重新定义 B01 已锁的鉴权基础设施；不画登录页（那由 `auth` feature 的 C04/C05 产出） |
| C03 · I | 信息架构师 | 本 feature 的功能清单 / 业务流程图 / 状态机 / 页面清单 / 导航结构 | 不出 URL / 路由、不画单页布局、不设计接口 |
| C04 · N | 交互设计师 | 单页布局 / 元素 / 行为 / 4 态 | 不定接口、不写代码、不出真实表名 / SQL / 接口路径 |
| C05 · H | 原型工 | 出可点通 HTML + 配套引导页 | 不动业务规范 |
| C06 · E | 产品经理 | 基于已稳定原型 + 上游沉淀输出 feature 级 PRD | 不新增需求、不改字段、不写代码 |
| D01 · D | 数据建模师 | 表结构、枚举、字段校验、索引 | 不定路由/接口、不定页面、不重定义 C03 状态机 |
| D02 · L | 路由/接口设计师 | 全局路由表、URL 规则、接口契约、错误码、并发策略 | 不写 SQL、不写 UI |
| D03 · V | QA / 文档审计师 | V01 上游链 / V02 模块闭环 / V03 PRD 回链 | 不写业务代码、不拆 Story、不打编码包 |

## 六、阶段闸门（Gate）

每阶段必经闸门才能进入下一阶段。任何 Gate 不通过 → 回到本阶段重做，**严禁强行往下推**。

### F · 一次性闸门（3 颗，全部 ✅ 才能开任何 feature 的 C）

| Gate | 通过判定 |
|------|---------|
| G-A | 架构规范无遗留问题、技术栈/目录/DB/API/编码规范全部锁死、surface 清单与鉴权基础设施锁死 |
| G-X | 体验定调有 ≥3 个参照、≥3 个反例、≥5 个可验证形容词 |
| G-S | 设计 Token 完备、所有组件都有 4 态、与 X 调性吻合、`prototype-style/` 可独立运行 |

### C · 按 feature 闸门（9 颗，逐 feature 跑）

| Gate | 通过判定 |
|------|---------|
| G-R | feature 需求基线无 `[待确认]`、需求清单 R-ID 唯一、用户签字 |
| G-P | 本 feature 的角色 × surface 矩阵闭合，已增量并入 `docs/C02-permissions/` 三份核心文件（01-roles / 02-authz-mechanism / 03-data-model），且与 B01 鉴权基础设施一致 |
| G-I | 本 feature 功能清单、流程图、状态机、页面清单均被 R-ID 全覆盖；状态机闭合 |
| G-N | 单页规范元素齐全、4 态齐全、引用 B03 设计系统对齐 |
| G-H | 原型可点通、4 态齐全、配套引导页（global / feature / surface）三级齐备且与设计系统风格一致；用户三轮内不再大改 |
| G-E | feature PRD 多文件齐；每句话可回链上游 ID；术语统一；99 节为空 |
| G-D | 数据规范字段齐、校验完整、覆盖 PRD 中所有数据形态；不重定义 C03 状态机 |
| G-L | 全局路由表与 C03 页面清单一一对齐；接口规范覆盖所有页面行为与状态转移、错误码完备、与 D 字段一一对齐 |
| G-V | V01/V02/V03 三份报告皆无红色项；所有 99 节为空；取得后**才**进入本 feature 开发 |

## 七、最小用法

```
[一次性] 读 A-framework/A00-04 建好 docs/ 目录
[一次性] B-foundation/B01 A → B02 X → B03 S → 全部 ✅
        【B03 双产出】docs/B03-design/design-system/（markdown 规范）
                     docs/B03-design/prototype-style/（可运行 CSS / JS 资产，C05 原型相对路径引用）

[按 feature 循环 A] for each <feature>:
  C-product/C01 R → C02 P → C03 I → C04 N → C05 H（一次性出齐 + 引导页）→ C06 E（多文件 PRD）
  ——到此可交付产品设计包，不必继续。

[按 feature 循环 B] 需要入开发时再走:
  D-develop/D01 D → D02 L → D03 V（V01+V02+V03 全绿）→ 进入开发
```

每一步只做一件事，AI 只看当下需要的文件。
