# A00-01 · 框架总览

## 定位

业务需求到开发规范的 AI 协同框架。**B（定调）→ C（产品）→ D（开发）** 三层、7 阶段。

---

## 系统优先原则

产出按「**系统 → 模块 → 功能**」三层组织。系统（如 user、admin、merchant）是独立产品单元，拥有独立设计、交互、原型和接口。

| 归属 | 阶段 |
|------|------|
| 共享 | B01（技术架构）、D01（数据模型） |
| 按系统 | B02（体验设计）、C01（需求权限）、C02（交互）、C03（原型）、D02（接口） |

---

## 7 阶段

### B 层 · 定调

| 阶段 | 归属 | 输出 |
|------|------|------|
| B01·A 技术架构 | 共享 | 选栈、目录、DB/API 规范、系统清单、鉴权 |
| B02·XS 体验设计 | 按系统 | 产品调性、设计 Token、组件、布局、原型样式包 |

### C 层 · 按系统×功能循环

| 阶段 | 输出 |
|------|------|
| C01·RP 需求权限 | 需求清单、用户故事、验收标准、权限矩阵 |
| C02·IN 交互 | 功能清单、流程图、状态机、页面清单、布局与行为 |
| C03·H 原型 | 可交互系统级闭环 HTML 原型 |

### D 层 · C 层完成后

| 阶段 | 归属 | 输出 |
|------|------|------|
| D01·D 数据 | 共享 | 表结构、枚举、校验、索引 |
| D02·L 接口 | 按系统 | 路由表、接口契约、错误码、并发策略 |

---

## 走序

```
B01·A(共享) → 各系统 B02·XS → B 层冻结

for each <功能>:
  for each <系统>:
    C01·RP → C02·IN → C03·H
  D01·D(共享)
  for each <系统>:
    D02·L
  → 开发
```

- B01 冻结 → 各系统 B02 可并行
- B 全冻结 → C 循环
- 各系统 C 链路互不阻塞
- D01 冻结 → D02 可并行
- 每轮循环围绕一个功能

---

## 步骤类型

- **2 步**（B01、B02、C01、C02）：用户输入 → AI 输出
- **1 步**（C03、D01、D02）：AI 基于上游直接输出

---

## 核心原则

| 原则 | 说明 |
|------|------|
| 系统优先 | 除 B01/D01 外一切按系统独立 |
| 设计先行 | 原型钉死形态，再倒推数据与接口 |
| 功能粒度循环 | C/D 层以功能为单位 |
| 增量融合 | 新功能产出增量融合到已有文档 |
| 上下文最小化 | AI 只看当前需要的上游 |

---

## 角色边界

| 阶段 | 严禁 |
|------|------|
| B01 | 不写功能字段、页面、业务角色枚举 |
| B02 | 不定接口、不定页面交互逻辑 |
| C01 | 不画流程图、不定状态机 |
| C02 | 不定接口、不写代码、不出表名 |
| C03 | 不动业务规范、不写后端代码 |
| D01 | 不定路由/接口、不定页面 |
| D02 | 不写 SQL、不写 UI |
