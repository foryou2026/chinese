# A00-01 · 框架总览

## 定位

从「业务需求」到「开发规范」的 AI 协同提示词框架。按 **B（定调）/ C（产品设计）/ D（开发规范）** 三层组织，共 7 阶段。每阶段 AI 按模板产出文档，所有产物可被下游机械消费。

---

## 系统优先原则

所有产出按「**系统(system) → 模块(module) → 功能(feature)**」三层组织。每个系统（如 user、admin、merchant）是独立产品单元，拥有独立的交互设计、原型和接口。

| 类型 | 归属 | 说明 |
|------|------|------|
| 必须共享 | `_shared/` | 数据模型（D01）、业务需求与权限（C01）、技术架构（B01）、设计系统（B02） |
| 必须独立 | `<system>/` | 信息架构与交互（C02）、HTML 原型（C03）、接口规范（D02） |
| 按需评估 | 视情况 | 部分 UX 可按系统定制，通过设计系统的系统变体支持 |

---

## 三层结构与 7 阶段

### B 层 · 一次性定调（项目级共享）

| 阶段 | 角色 | 输出 |
|------|------|------|
| B01·A 技术架构 | 架构师 | 选栈、目录、DB/API/编码规范、系统清单、鉴权基础设施 |
| B02·XS 体验与设计 | 体验总监+UI | 产品调性 + 设计 Token、组件、布局、原型样式包 |

### C 层 · 按系统×功能循环

| 阶段 | 角色 | 归属 | 输出 |
|------|------|------|------|
| C01·RP 需求与权限 | 需求分析师+权限 | **共享** | 需求清单 R-ID、用户故事、验收标准、角色权限矩阵 |
| C02·IN 架构与交互 | 信息架构师+交互 | **按系统** | 功能清单、流程图、状态机、页面清单、单页布局与行为 |
| C03·H HTML 原型 | 原型工程师 | **按系统** | 可交互跳转的系统级闭环 HTML 原型 |

### D 层 · C 层完成后按系统×功能循环

| 阶段 | 角色 | 归属 | 输出 |
|------|------|------|------|
| D01·D 数据规范 | 数据建模师 | **共享** | 表结构、枚举、字段校验、索引 |
| D02·L 接口规范 | 接口设计师 | **按系统** | 路由表、接口契约、错误码、并发策略 |

---

## 走序

```
[一次性] B01·A → B02·XS → B 层冻结

[按功能循环]
  for each <功能>:
    C01·RP（共享）→ 
    for each <系统> involved:
      C02·IN → C03·H → D02·L
    D01·D（共享）→ 进入开发
```

- B 层冻结后才能开始 C 循环
- C01 冻结后各系统 C02 可并行
- D01 冻结后各系统 D02 可并行
- C03 稳定后启动 D 层
- 每轮循环围绕**一个功能**（如 `account-entry`、`learning`）

---

## 核心原则

| 原则 | 说明 |
|------|------|
| 系统优先 | 产出按"系统→模块→功能"三层组织 |
| 设计先行 | 原型钉死产品形态，再倒推数据与接口 |
| 功能粒度循环 | C/D 层以功能为单位循环 |
| 共享沉淀 | 数据模型、业务规则、设计 Token 跨系统共用 |
| 增量融合 | 新功能产出增量融合到全局文档 |
| 上下文最小化 | 每阶段 AI 只看当下需要的上游文件 |

---

## 角色边界

| 阶段 | 严禁 |
|------|------|
| B01 架构师 | 不写功能字段、页面、业务角色枚举 |
| B02 体验+设计 | 不定接口、不定页面交互逻辑 |
| C01 需求+权限 | 不画流程图、不定状态机、不出 mermaid |
| C02 架构+交互 | 不定接口、不写代码、不出表名/SQL |
| C03 原型 | 不动业务规范、不写后端代码 |
| D01 数据 | 不定路由/接口、不定页面 |
| D02 接口 | 不写 SQL、不写 UI |

---

## 步骤说明

- **2 步阶段**（B01、B02、C01、C02）：用户输入 → AI 输出
- **1 步阶段**（C03、D01、D02）：AI 基于上游冻结产物直接输出

## 文件命名

| 对象 | 规则 | 示例 |
|------|------|------|
| 框架层 | `A00-<序号>-<名称>.md` | `A00-01-框架总览.md` |
| 阶段模板 | `<阶段码>-<步骤号>-<类型>-<名称>.md` | `B01-01-用户输入-技术偏好.md` |
| system ID | 全小写，项目内唯一 | `app`、`admin`、`merchant` |
| 模块 ID | 全小写短横分隔 | `auth`、`course`、`order` |
| 功能 ID | 全小写短横分隔 | `account-entry`、`learning` |

## 冻结定义

- 文件头 `冻结状态` 标为 `已冻结` = 内容稳定，下游可引用
- 前提：§99 待确认问题一节为空
- 冻结后仅可增量追加（标注 `[本轮新增]` / `[本轮变更]`），不删改已有内容
- B 层全冻结 → 启动 C 循环；C01 冻结 → C02 并行；C02→C03 冻结 → 启动 D 层

---

## C/D 层共享与独立协作

```
C01 共享需求 → _shared/C01-requirements/<module>/<feature>/
                    ↓ 被各系统引用
C02 系统A   → <systemA>/C02-ia-interaction/<module>/<feature>/
C02 系统B   → <systemB>/C02-ia-interaction/<module>/<feature>/
                    ↓
C03 系统A   → C03-prototype/<systemA>/<module>/<feature>/
C03 系统B   → C03-prototype/<systemB>/<module>/<feature>/
                    ↓
D01 共享数据 → _shared/D01-data/<module>/<feature>/
                    ↓ 被各系统引用
D02 系统A   → <systemA>/D02-api/<module>/<feature>/
D02 系统B   → <systemB>/D02-api/<module>/<feature>/
```
