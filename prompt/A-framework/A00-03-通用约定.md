# A00-03 · 通用约定

> 所有阶段、所有模板共用。AI 在任何阶段开口前必须先读本文件。

---

## 一、AI 输出文件的通用骨架

每份 AI 输出文件必须以此头部开始：

```markdown
# <文件名>

> **阶段**：<B01-A / B02-XS / C01-RP / C02-IN / C03-H / D01-D / D02-L>
> **角色**：<阶段对应角色>
> **归属**：<共享(_shared) / 按系统(<surface-id>)>
> **系统**：<surface-id>（共享阶段写「全系统」；B 层写「全局」）
> **模块**：<模块 ID>（B 层写「全局」）
> **功能**：<功能 ID>（B 层写「全局」）
> **上游依赖**：<本次实际读取的上游文件路径>
> **冻结状态**：未冻结 / 已冻结

---

## 0. 摘要（3-5 行）
<本文件最关键结论>

## 1. 正文 ...

## 99. 待确认问题
- [ ] 编号：<问题>（影响：<下游文件>）
```

> 99 节为空 = 可冻结；非空 = 必须补答后才能冻结。

---

## 二、Mermaid 规范

- 一律用 `mermaid` 代码块，禁止上传图片
- 主路径用 `flowchart LR`；状态机用 `stateDiagram-v2`；时序用 `sequenceDiagram`；ER 用 `erDiagram`
- 节点 ID 用大写字母+数字（A1/B2），可读名称写 `[]` 内
- 正常路径用 `-->`，异常路径用 `-.->`
- 每张图后补一段文字说明，确保不依赖颜色也能读懂
- 兼容 Excalidraw 的 Mermaid 导入格式

---

## 三、命名约定

| 对象 | 规则 | 示例 |
|------|------|------|
| surface ID | 全小写单词，项目内唯一 | `app`、`admin`、`merchant` |
| 模块 ID | 全小写短横分隔，顶层业务域 | `auth`、`course`、`order` |
| 功能 ID | 全小写短横分隔，模块内子功能 | `account-entry`、`learning`、`checkout` |
| 需求 ID | `R-<模块>-<seq3>` | `R-auth-007` |
| 模块 ID | `M-<模块>-<seq3>` | `M-auth-012` |
| 状态机 ID | `SM-<模块>-<seq2>` | `SM-auth-01` |
| 页面 ID | `P-<surface>-<模块>-<seq3>` | `P-app-auth-012`、`P-admin-course-001` |
| 接口 ID | `API-<surface>-<模块>-<verb>-<noun>` | `API-app-auth-create-session` |
| 角色 ID | `ROLE-<NAME>` | `ROLE-USER`、`ROLE-ADMIN` |
| 表名/字段 | 按架构规范定义（通常 `snake_case`） | `order_items` |

> **页面 ID 与接口 ID 必须带 surface 前缀**，确保跨系统不撞号。
> 需求 ID（R-ID）不带 surface 前缀，因为需求是共享的。

## 四、输出落盘路径约定

所有 AI 产出文件落盘到 A00-01 §九定义的全局输出目录。按**归属**分为共享路径和系统路径：

### 共享产出路径

| 阶段 | 落盘根路径 |
|------|----------|
| B01 | `docs/B01-architecture/` |
| B02 | `docs/B02-experience-design/` |
| C01 | `docs/_shared/C01-requirements/<module-id>/<feature-id>/` |
| D01 | `docs/_shared/D01-data/<module-id>/<feature-id>/` |

### 按系统独立产出路径

| 阶段 | 落盘根路径 |
|------|----------|
| C02 | `docs/<surface-id>/C02-ia-interaction/<module-id>/<feature-id>/` |
| C03 | `C03-prototype/<surface-id>/<module-id>/<feature-id>/` |
| D02 | `docs/<surface-id>/D02-api/<module-id>/<feature-id>/` |

### 权限文件（共享，按端拆分）

```
docs/_shared/C01-requirements/permissions/
  roles.md                    # 角色枚举
  app-matrix.md               # app 端权限矩阵
  admin-matrix.md             # admin 端权限矩阵
```

> C03 直接位于项目根目录下（非 docs/ 内），确保 `cd C03-prototype && python -m http.server` 可直接运行。

## 五、引用规则

- 仓内文件：用相对路径 markdown 链接
- 同文件内章节：`见本文 §3.2`
- 上游冻结文件：必须出现在本文件头部「上游依赖」字段
- **跨系统引用**：按系统独立阶段（C02/C03/D02）不得引用其他系统的同阶段产出；可引用共享阶段（C01/D01）的产出
- 跨功能引用：原则上不允许；如确需，先沉淀到共享层再由双方引用

## 六、AI 自检 Checklist

每次输出前内部跑一遍，任何一项 No 则重写：

- [ ] 是否只读了系统消息列出的上下文？（禁止偷看未授权文件）
- [ ] 输出文件头部元信息是否齐全？（含系统、模块、功能字段）
- [ ] 产出文件是否落盘到正确的路径？（共享 → `_shared/`；按系统 → `<surface>/`）
- [ ] 是否有 `99 待确认问题` 一节？
- [ ] 是否越界做了本角色不该做的事？（参照 A00-01 角色边界表）
- [ ] 文件是否超长？（超 1200 行必须拆分）
- [ ] 是否塞入了用户没要求的东西？
- [ ] 页面 ID / 接口 ID 是否带了 surface 前缀？

---

## 七、严禁项

1. 在用户没说的地方擅自做技术决策（应记入 99 节待确认）
2. 一次输出多份不同阶段的文件
3. 把上下文窗口塞满整个仓库
4. 用图片代替 mermaid
5. 越界（需求阶段写表结构、数据阶段写接口、交互阶段写代码......）
6. 在 B 层未全部冻结时启动任何功能的 C 循环
7. **按系统独立阶段跨系统引用同阶段产出**（如 app 的 C02 引用 admin 的 C02）
8. **共享阶段产出放到系统目录下**（如 D01 数据规范放到 `app/D01-data/`）

---

## 八、形容词具象化原则

任何「风格 / 体验 / 感觉」类形容词必须配套：

- 至少 1 个参照（产品名 + 截图链接 + 借鉴点）
- 至少 1 个反例（「不要像 X 那样」）
- 至少 1 个可验证维度（如：圆角 <= 6px、动效 < 200ms、留白比例 >= 30%）

否则该形容词视为无效，需补充具象化描述。
