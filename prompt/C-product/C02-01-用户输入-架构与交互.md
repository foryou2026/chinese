## 一、信息架构方向

### 1. 功能模块粗描（产品视角）

> 用户能感知到的"功能块"，不是代码模块。一行一条，每条标"主/次"。
> **引导**：模块边界是否清晰？有无重叠？每个模块对各角色是否可见/可写？

```
[例：]
- (主) 课程浏览：用户找到并查看课程
- (主) 订单与支付：把课程"装进我的"
- (次) 收藏与分享
```

### 2. 关键业务流程（用大白话写完整步骤）

> 至少 1 条主流程 + 1 条异常路径。AI 会画 mermaid。
> **引导**：每个分叉点都要说清楚——判断条件是什么？失败/超时/无权限/重复操作怎么办？

```
[主流程：XXX]
1. 用户做什么
2. 系统判断什么 → 分叉条件
3. 成功路径
4. 失败路径
```

```
[异常流程 1：XXX]
- ...
```

### 3. 业务对象的状态流转（状态机）

> 每个有状态的"东西"列一份。
> **引导**：请明确——状态枚举是否齐全？每个转移的触发者是谁（角色/系统/定时）？是否有终态？是否允许回退？

```
[例：订单状态]
- 状态枚举：待支付 / 已支付 / 已完成 / 已退款 / 已取消
- 待支付 →（支付成功）→ 已支付
- 终态：已完成 / 已退款 / 已取消
- 不可回退：已支付 不可回到 待支付
```

### 4. 你已经想好的页面清单（如有）

> 不要写 URL，只写"页面名 + 一句话职责 + 主要服务的角色"。
> **引导**：列表/详情/编辑三件套是否统一形态？弹窗 vs 抽屉 vs 新页的决策依据？

```
1. 首页：用户进来看到的第一屏，承载推荐与导航
2. 课程列表：按分类/搜索找课程
3. ...
```
