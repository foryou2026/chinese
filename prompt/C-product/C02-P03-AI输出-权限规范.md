```markdown
# 角色定义

## 角色枚举

| 角色 ID | 显示名 | 来源（B02 用户输入条目）| 优先级 |
|--------|-------|---------------|--------|
| ROLE-USER | 普通用户 | ... | 1 |
| ROLE-EDITOR | 内容编辑 | ... | 50 |
| ROLE-ADMIN | 超管 | ... | 99 |

## 多角色策略
- 一个用户可拥有多个角色：是 / 否
- 角色冲突时按"优先级"取最高

## 角色 × surface 矩阵（必须存在；单端项目仅一列）

> 引自 B01 `08-surfaces.md` 的 surface-id 列表。同一角色出现在多个 surface 时，下游 `<surface>-auth` feature 各自实现登录入口，但**共享** users 表与本表角色定义。

| 角色 ID | `app` | `admin` | `merchant` | `open` |
|---------|-------|---------|-----------|--------|
| ROLE-USER | ✅ 主 | ❌ | ❌ | ❌ |
| ROLE-EDITOR | ❌ | ✅ 主 | ❌ | ❌ |
| ROLE-ADMIN | ❌ | ✅ 主 | ✅ 代登 | ❌ |

## 全局权限矩阵（页面/功能点级）

| 资源 / 操作 | ROLE-USER | ROLE-EDITOR | ROLE-ADMIN |
|------------|-----------|-------------|-----------|
| 公开页 | ✅ | ✅ | ✅ |
| 个人中心 | ✅(自己) | ✅(自己) | ✅(任意) |
| 课程管理 | ❌ | ✅ | ✅ |
| 用户管理 | ❌ | ❌ | ✅ |
| 报表 | ❌ | ❌ | ✅ |

> 具体到接口的权限校验，落到 L 阶段每个接口节内引用本表。

## 数据可见范围原则

| 角色 ID | 默认可见范围 | 说明 |
|---------|-----------|------|
| ROLE-USER | owner-only | 只看自己创建的 |
| ROLE-MERCHANT | own-shop | 只看自己 shop_id |
| ROLE-ADMIN | global | 全平台 |

## 系统级硬约束
- ROLE-ADMIN 不能删除自己
- 被禁用账号在 authz 层统一返回 403 + 错误码 `AUTH_DISABLED`（具体提示文案在 `<surface>-auth` feature 中）
```

```markdown
# 授权校验机制

> Token / 会话 / 多设备 / 刷新 一律引用 B01 `09-auth-infra.md`，本文件不重述。

## 前端
- 路由守卫：无权限页面如何拦截（统一组件，不同 surface 复用）
- 菜单过滤：按角色 + surface 双维度
- 按钮级控制：隐藏 vs 禁用 的统一规则
- Token 过期处理：统一拦截器 + 触发 `<surface>-auth` feature 的刷新流程

## 后端
- 中间件 / 拦截器
- 校验顺序：鉴权（Token 解析，B01 09-auth-infra）→ surface 校验（aud claim）→ 角色 → 资源所属 → 业务
- 角色信息来源：Token claim `roles[]` + DB 校对
- 越权统一返回：HTTP 403 + 业务错误码 `AUTHZ_DENIED` + 不暴露原因

## 行级 / 字段级（如需）
- 数据库行级安全策略（PG RLS / 应用层 Service 过滤）
- 字段级脱敏规则（按角色返回不同字段子集）

## 审计日志
- 必记字段：操作者 user_id、surface、时间、IP、UA、动作、目标资源、结果
- 存储位置、保留时长（与 B01 09-auth-infra §4 联动，登录类事件由 09-auth-infra 写入；业务越权事件由本机制写入；同表 `audit_logs` + 类型字段）
- 谁能查询（按 `01-roles.md` 中角色×surface 矩阵决定）
```

```markdown
<!-- TARGET-PATH: docs/C02-permissions/04-auth-feature-guideline.md -->

# auth 类 feature 编写指南（仅当项目存在登录/注册/忘密交互时输出本文件，否则整份文件留 "N/A" 即可）

> **背景**：本框架不在 B 阶段规定任何登录页/注册页/忘记密码流程；这些都是 feature 内部交互。如果项目存在登录需求，**作为一个普通 feature**（推荐 id `auth`，命名规则与其他 feature 完全一致，无任何强制前缀/后缀）走 C01→C05 循环。多端共用时按 `<feature>/<surface>/` 通用多端形态拆子目录——**不要**额外造 `<surface>-auth` 这种带 surface 前缀的 feature。

## 1. 该 feature 必须满足的硬约束

- Token / 密码哈希 / OAuth provider / MFA：**全部**复用 B01 `09-auth-infra.md` 的全局决策，不许自定
- 角色枚举：只能从 B02 `01-roles.md` 已声明的角色中映射
- 后端路由前缀：单端 `/api/auth/*`；多端 `/api/<surface>/auth/*`
- token claim：必须包含 `aud=<surface>`、`roles[]`、`user_id`
- 审计：登录/登出/Token 刷新/失败必写 `audit_logs`，与 B02 `02-authz-mechanism.md §审计` 共表

## 2. 该 feature 自由决定的部分

- 登录字段（手机号 / 邮箱 / 用户名 / SSO 入口）
- 是否开放自助注册 / 邀请制 / 仅后台开通
- 忘记密码恢复方式（邮箱链接 / 短信验证码 / 联系管理员）
- 登录页 UI、文案、品牌
- 登录失败提示文案（统一错误码 `AUTH_*`，文案 feature 自定）
- 该端是否启用 MFA / 第三方登录（在 B01 09-auth-infra §3 已声明的 provider 范围内挑选）

## 3. 该 feature 走 C 循环时的输出位置

- 多端项目（多个 surface 都要登录）：`docs/C01-requirements/auth/baseline.md`（业务整体单份）+ `docs/C03-ia/auth/<surface>/...`、`docs/C04-pages/auth/<surface>/...`、`docs/C05-prototype/auth/<surface>/...`、`docs/C06-prd/auth/<surface>/...`
- 单端项目：`docs/C0[1-5]-*/auth/...`（按非 surface 形态走）
- 仅一个 surface 需要登录：依然只在该 surface 的子目录下出产物，不另起 feature

## 4. 校验项（D03 验证阶段使用）

- [ ] feature 内未私自重定义 token 类型 / 哈希算法
- [ ] feature 路由单端落 `/api/auth/*`、多端落 `/api/<surface>/auth/*`
- [ ] 涉及的角色全部存在于 B02 `01-roles.md`
- [ ] 该角色在该 surface 上 ✅ 出现于 `01-roles.md` 角色×surface 矩阵
```
