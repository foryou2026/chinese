## 本阶段做什么、不做什么

| ✅ 本阶段产出 | ❌ 本阶段不做 |
|------|------|
| 全局角色枚举（项目只一份）| 登录页 / 注册页 / 忘记密码页 的字段、交互、文案 |
| 角色 × surface 矩阵（哪些角色出现在哪些端）| 登录方式选型（手机号/邮箱/微信）——那是 feature 级 |
| 多角色与角色组策略 | Token 类型/刷新/存储 —— B01 09-auth-infra 已定 |
| 授权机制（前端守卫 / 后端拦截 / 行级）| 账号过期、被锁、推送重设链接 之类交互细节 |
| 数据可见范围（行级/字段级的原则）| |
| 审计日志、合规、用户/角色数据模型骨架 | |

> **「登录」不是全局事**：如果项目有登录/注册/忘密这类交互，它们属于对应的 feature 内部交互（例如名为 `auth` 的 feature），与「商品」「订单」是同类别的普通 feature，按通用规则走 C01→C05 循环；多端共用时按 `<feature>/<surface>/` 拆子目录。**B02 本身永不画登录/注册/忘密流程**。如果项目根本没有登录需求，则不存在该类 feature，整个流程同样照常。

## 1. 系统有哪些角色

> 用大白话写每种角色是谁、能做什么级别的事。角色是**全局**枚举，跨 surface 共享（同一个 `ROLE-ADMIN` 可能的用户同时在 admin 与 merchant 出现）。

```
[例：]

1. 老板（最高权限）
   - 看所有数据，做所有操作
   - 管理员工账号
   - 看财务报表

2. 运营（日常操作者）
   - 处理业务数据（订单/商品/库存）
   - 不能看财务
   - 不能管理其他账号

3. 客服（受限）
   - 只读，能登记售后

4. 终端用户
   - 使用业务
```

## 1.5 角色 × 应用端（surface）矩阵

> 从 B01 08-surfaces.md 抵 surface-id 列表。标识每个角色**在哪些端出现**（不是"能不能登"，而是"这个端为谁设计"）。同一个角色可出现在多个端。

| 角色 ID | `app` | `admin` | `merchant` | `open` |
|---------|-------|---------|-----------|--------|
| ROLE-USER | ✅ 主 | ❌ | ❌ | ❌ |
| ROLE-MERCHANT | ❌ | ❌ | ✅ 主 | ❌ |
| ROLE-OPS | ❌ | ✅ 主 | ❌ | ❌ |
| ROLE-ADMIN | ❌ | ✅ 主 | ✅ 可代登 | ❌ |
| ROLE-API-CLIENT | ❌ | ❌ | ❌ | ✅ |

> "代登"场景（admin 代商家登录调查）请在备注明确。
