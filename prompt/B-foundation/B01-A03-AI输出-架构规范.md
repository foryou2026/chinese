```markdown
# 应用端清单与跨端隔离策略

> 本文件钉死本项目有几个**应用端**（surface）、每端的用户群与边界。
> 单端项目仅列 1 行；多端项目所有 C/D 阶段产物按本表的 surface-id 分子目录。

## 1. surface 清单

| surface-id | 显示名 | 用户群 | 主要鉴权方式（粗）| 部署形态 | 域名 / 路径 | 是否本期上线 |
|------------|-------|-------|-----------------|---------|-----------|------------|
| `app` | 用户端 | 终端用户 | 账密 + OAuth | Web SPA | app.x.com | ✅ |
| `admin` | 管理后台 | 内部运营 | 账密 + MFA | Web SPA（内网）| admin.x.com | ✅ |
| `merchant` | 商家端 | 入驻商家 | 账密 | Web SPA | merchant.x.com | 二期 |

> **单端项目模板**：仅一行 `app | 用户端 | 终端用户 | 账密 | Web SPA | x.com | ✅`，下游目录全部按非 surface 形态走（不出现 `<surface>/` 子目录）。

## 2. 跨端隔离策略

| 维度 | 策略 |
|------|------|
| 前端代码库 | Monorepo 下 `apps/<surface>/` 各一个工程；共享组件走 `packages/ui` |
| 用户体系 | 共享 users 表，按 `user_roles.surface` 字段隔离角色域；同一 user 可在多端拥有不同角色 |
| API 路由前缀 | `/api/<surface>/*`（如 `/api/app/products`、`/api/admin/products`），即使逻辑相同也分前缀以便鉴权策略隔离 |
| 鉴权域 | 每个 surface 独立 cookie domain / 独立 token claim `aud=<surface>`；不允许跨 surface 复用 token |
| 部署 | 每个 surface 独立前端构建产物；后端可单体也可 BFF |

## 3. 跨端业务关联（强关联场景）

> 例：admin 配置商品 → app 看到上架；admin 改价 → app 实时刷新。本节列出本项目的强关联场景，作为 C 阶段做"跨端 feature"决策的参考。

| 关联场景 | 触发端 | 受影响端 | 落点 |
|---------|-------|---------|------|
| 商品配置 → 商品展示 | admin | app, merchant | 共享 `product` feature 的 D01 数据 + 共享 SM-product 状态机 |
| 订单售后 → 商家通知 | app | merchant, admin | 共享 `order` feature；事件总线 |

## 4. surface × auth feature 矩阵

> B02 已抹除"登录流程"；每个 surface 必须有自己的 auth feature 走 C 循环。

| surface-id | 对应 auth feature | feature 路径 |
|------------|------------------|-------------|
| `app` | `app-auth` | docs/C01-requirements/app-auth/ |
| `admin` | `admin-auth` | docs/C01-requirements/admin-auth/ |
| `merchant` | `merchant-auth` | docs/C01-requirements/merchant-auth/（二期）|
```

```markdown
| 项目 | 决定 |
|------|------|
| Token 类型 | JWT / Opaque |
| Access Token 有效期 | |

| 刷新机制 | 静默刷新 / 用户感知 / 滚动刷新 |
```
