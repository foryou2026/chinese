# 40 · F 阶段（功能开发）编排与澄清增强

> **角色**：PM + Architect 合议
> **目标**：逐功能产出 F1（数据）→ F2（接口）→ F3（页面交互）三份规范。
> **粒度**：一次只做一个功能（feature），跑完它的 F1→F2→F3 再做下一个。

## 二、本框架补的澄清环节

每个 Fx 子阶段：用户填表 → **AI 必须先出澄清** → 用户答 → AI 出最终规范。

```
Step 1 你填 F<X>-用户 → function/<proj>/<feat>/_input/F<X>-用户.md
Step 2 AI 按 §四 模板出 F<X> 澄清问题
       → function/<proj>/<feat>/_questions/F<X>-round1.md
Step 3 你答完 → AI 按 F<X>-AI 模板出规范
       → function/<proj>/<feat>/F<X>-...规范.md
```

闸门：每份 F 输出文件末尾"99 待确认"为空才允许进入下一个 F 子阶段。

## 五、各 F 子阶段必盘点维度

### F1 必盘点 12 维
1. 数据对象边界（一张表 vs 多张表）
2. 主键策略（uuid / 雪花 / 自增）与 G1 是否一致
3. 通用字段（created_at/updated_at/deleted_at/created_by）
4. 枚举字段是否需要独立表（i18n 友好）
5. 关系类型（1-1 / 1-N / N-N）与外键策略
6. 状态字段是否需要状态机表 + 流转日志
7. 计算字段（冗余 vs 视图 vs 实时算）
8. 索引建议（查询模式驱动）
9. 唯一约束与并发冲突策略
10. 校验规则（DB 层 / 应用层 / 前端层 各管什么）
11. 历史/版本（是否需要 history 表）
12. 数据量级与分区/归档策略

### F2 必盘点 14 维
1. 接口集合（按资源 / 按用例划分）
2. URL 与 HTTP 方法（与 G1 §API 规范对齐）
3. 入参（query/path/body）的位置约定
4. 响应结构（统一 envelope vs 直接 body）
5. 错误码体系（与 G1 错误码规范对齐）
6. 校验顺序（鉴权 → 参数 → 业务）
7. 权限校验（与 G3 角色矩阵对齐）
8. 幂等策略（写接口）
9. 分页/排序/筛选标准
10. 批量操作 / 长任务（同步 vs 异步 + 轮询/SSE/WebSocket）
11. 缓存（HTTP 缓存头 / 服务端缓存）
12. 限流 / 反爬（每用户每接口 QPS）
13. 审计日志（哪些写接口必记）
14. 失败重试与补偿（与 mock 第三方关联）

### F3 必盘点 12 维
1. 页面 ↔ P4 HTML 文件对照
2. 每个元素的数据来源（F1 字段）
3. 每个交互的接口调用（F2 接口 ID）
4. 表单的字段顺序、必填、校验文案（中英多语言）
5. 列表的分页/排序/筛选默认值与持久化（URL 参数 / localStorage）
6. 4 态实现（empty/error/loading/forbidden）的精确触发条件
7. 权限控制（哪些按钮在何角色下灰显/隐藏）
8. 跨页面跳转的参数传递与状态保持
9. 离开未保存提示
10. 键盘可达性（焦点顺序、快捷键）
11. 国际化（文案占位 + 数字/日期/货币本地化）
12. 场景验证脚本（典型用户故事的端到端用例）

## 七、闸门 G-F 判定（按 feature 单独过）

- F1/F2/F3 三份输出 §99 都空
- 三份文件之间引用一致：
  - F2 接口的入参/出参字段都能在 F1 找到
  - F3 中每个数据字段都能在 F1 找到
  - F3 中每个 API 调用都能在 F2 找到
  - F3 中每个权限判断都能在 G3 角色矩阵找到
- F3 §场景验证 至少 5 条，全部"期望结果"明确
- 该功能 F1+F2+F3 都签字 `已冻结 v1`

> 通过后该功能进入 V 阶段；其他功能可并行做 F。
