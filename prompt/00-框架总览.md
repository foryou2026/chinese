# 00 · 框架总览：自建 Skills 提示词体系

> **本框架是什么**
> 一套从「需求」到「开发前最后一公里」的提示词模板集。它不是 BMAD/GSTack 本身，而是把它们最有价值的工程化思维提炼后，结合本仓库 `/grules`、`/template`、`/function` 的现有约定，重组成「你能完全自控」的 Skills 体系。
>
> **谁来用**
> - 你（产品经理 / 全栈）：每一步只负责"填表"和"审核"。
> - AI（Copilot / Claude / Gemini 等）：每一步只在指定上下文里做指定动作，不自由发挥。
>
> **核心承诺**
> 1. 每一份"用户输入模板"都对应一份"AI 必须先发的澄清提问清单"，**问清楚之前 AI 不许出报告**。
> 2. 每一份"AI 输出模板"都有强格式约束，AI 必须按字段填，便于后续机器/人工审核。
> 3. 所有阶段以**文件**作为交接物，AI 不依赖记忆——下游只看上游产物文件。
> 4. 单文件 ≤ 1200 行，超出强制拆分。

## 一、整体阶段图

```
 ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
 │ R 需求    │ →  │ P 原型    │ →  │ G 全局规范│ →  │ F 功能开发│ →  │ V 开发前  │
 │ Analyst   │    │ Designer  │    │ Architect │    │ PM+Arch  │    │ SM + QA   │
 └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
      │              │                │                │                │
      ▼              ▼                ▼                ▼                ▼
  R3 需求基线    P3 IA + P4 HTML   G1/G2/G3 规范    F1/F2/F3 规范    V1 一致性报告
                                                                     V2 Story 拆分
                                                                     V3 编码上下文包
```

每一阶段的最终交付物会被**冻结**为下一阶段的唯一上游输入，AI 不得越级查找。

---

## 二、五阶段角色（从 BMAD 提炼，非照搬）

| 阶段 | 角色名 | 单一职责 | 严禁做的事 |
|------|--------|---------|-----------|
| R | **Analyst（需求分析师）** | 把模糊业务诉求 → 结构化需求清单 + 流程图 | 不做技术选型、不画 UI |
| P | **Designer（产品设计师）** | 信息架构 + HTML 静态原型 + 交互骨架 | 不写真实接口、不定数据库字段 |
| G | **Architect（架构师）** | 一次性沉淀 G1/G2/G3 全局规范 | 不写具体功能字段 |
| F | **PM + Architect 合议** | 逐功能产出 F1（数据）→ F2（接口）→ F3（页面） | 不一次产出多个功能 |
| V | **SM（Scrum Master）+ QA** | 跨文件一致性校验、拆 Story、打编码上下文包 | 不直接写业务代码 |

> **每个角色独立对话窗口**。开新对话时把"角色定义 + 阶段说明 + 上游产物"作为系统消息塞入，禁止跨角色串场。

## 三、每一步的"三件套"模式（从 GSTack 思想提炼）

每一步都是同一个三件套：

```
  ┌─────────────────────┐    ┌─────────────────────┐    ┌─────────────────────┐
  │ ① 用户输入模板        │ →  │ ② AI 澄清提问模板    │ →  │ ③ AI 输出模板        │
  │  user-input.md       │    │  ai-questions.md    │    │  ai-output.md       │
  └─────────────────────┘    └─────────────────────┘    └─────────────────────┘
```

**关键铁律**：

- 用户提交 ① 后，AI **第一轮回复必须是 ②**，把所有不清楚的点穷尽列出（编号、可选项、推荐项、影响范围）。
- 用户回复 ② 后，AI 才允许产出 ③。
- ③ 必须以 Markdown 文件保存到对应路径，不允许只在对话里"念一遍"。

> 这条铁律解决了"AI 一次塞太多信息然后丢细节"和"用户以为讲清楚了其实没讲清楚"两个最常见翻车点。

## 五、和已有资产的关系

| 已有资产 | 在本框架的角色 |
|---------|--------------|
| `/grules/G1~G3` | 知语项目当前已落地的全局规范实例（G 阶段产物范例） |
| `/template/all_totals/G1~G3, F1~F3` | 本框架 G/F 阶段的"用户输入"和"AI 输出"基础模板，已经成熟，**继续使用，不重写**，本框架在 30-/40- 文件中只补充"澄清提问"环节 |
| `/template/g_prompt/`、`/template/f_example/` | 现有用户输入样例 |
| `/function/` | 新功能产物落盘位置 |
| `/content/` | 内容/数据规范产物落盘位置 |

> 也就是说，本框架真正"新增"的内容是：**R 阶段、P 阶段、V 阶段**，以及**所有阶段的"AI 澄清提问"模板**。已有 G/F 体系不动。

## 六、跨阶段冻结规则

```
R3 通过 → 写入 content/<project>/requirements/baseline.md → R 阶段冻结
P3+P4 通过 → 写入 content/<project>/prototype/                → P 阶段冻结
G1/G2/G3 通过 → 写入 grules/<project>/                         → G 阶段冻结
F1/F2/F3 通过 → 写入 function/<project>/<feature>/             → F 阶段冻结（按功能）
V1/V2/V3 通过 → 写入 function/<project>/<feature>/_v/           → 进入编码
```

冻结后再改：必须新开一轮"变更说明"对话，AI 必须先生成 diff 报告再改文件。

---

## 七、最小用法（5 步速读）

1. 复制 `11-R1` 模板，填业务诉求 → 发给 AI（系统消息塞 `00`、`02`、`10`、`12`、`13`）。
2. AI 回澄清问题（来自 `12`）→ 你回答 → AI 输出 `13` 格式的需求基线。
3. 复制 `21-P1` 模板，填原型方向 → 进入 P 阶段，迭代 HTML 原型直至冻结。
4. 复制 `30/40` 编排文件中指定的 G/F 模板，逐项做。
5. 全部 F 完成后，进入 `50-V` 做一致性校验、Story 拆分、编码上下文打包。然后才允许写代码。

> 从这一刻起，AI 只在"打包好的小盒子"里做事。
