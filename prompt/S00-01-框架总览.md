# 01 · 框架总览

> 本文件回答三个问题：本框架是什么、为什么这样设计、怎么使用。

---

## 一、是什么

一套从「业务诉求」到「上线增长」全链路的标准化提示词模板集，覆盖 12 个阶段、12 种角色、约 43 份模板。

它不替你写代码，它让 AI 在每一步都按你定义的格式做事，并在所有不清楚的地方主动问你，直到产出可以被下游机械消费的文档。

---

## 二、为什么这样设计

AI 协同开发最常见的三个翻车点：

1. **AI 凭空脑补**：用户没说的细节，AI 直接默认了，下游全错。
2. **上下文塞太多**：一次喂 1 万字，AI 抓不住重点，丢字段。
3. **角色串场**：让"画原型的 AI"同时定数据库字段，结果两头不到位。

本框架的三个解药：

| 解药 | 落地形态 |
|------|---------|
| 强制澄清协议 | 每一步必经"用户填表 → AI 必先提问 → 用户答 → AI 出报告"，AI 跳步即打回 |
| 上下文最小化 | 每一步明确"系统消息只塞这几份文件"，禁止全塞 |
| 角色独立窗口 | 12 阶段 × 不同角色，独立对话窗口，互不污染 |

---

## 三、12 阶段执行顺序与依据

| 顺序 | 阶段 | 为什么这个时机 |
|------|------|--------------|
| 1 | R 需求分析 | 万事之始，把"模糊的话"变成结构化需求清单 + 流程图 |
| 2 | A 技术架构 | 选栈、定目录、定数据库/接口/编码规范——后续所有产物都按它走 |
| 3 | P 权限与角色 | 角色枚举要先定，所有功能的可见性、操作权限都要它 |
| 4 | D 数据模型（按功能） | **先于交互**：数据结构是地基，没数据谈不了接口和页面 |
| 5 | L 业务与接口（按功能） | 数据有了，定义对它能做什么操作 |
| 6 | X 体验定调（UX 调性） | **先于视觉**：决定"产品给人的感觉"，避免最终原型平庸 |
| 7 | S 设计系统 | 把 X 的调性落成 Token、组件、布局——全局视觉语言 |
| 8 | I 信息架构 | 路由结构、页面清单、导航结构，全局骨架 |
| 9 | N 页面交互（按页面） | 数据/接口/视觉/IA 全到位，才能精准描述每个页面的行为 |
| 10 | H HTML 原型 | 此时原型不是"瞎画"，而是把前面所有规范"活"出来——可点通、有真数据型态、有调性 |
| 11 | V 一致性校验 | 跨文件机械核对：字段、ID、引用是否一一对齐 |
| 12 | E SEO + GEO 营销优化 | 把上游所有沉淀转成 SEO + GEO 营销执行包：关键词、技术 SEO、Schema、llms.txt、AI 引擎、被引平台、UGC、监测、90 天计划 |

> V 阶段产出三份文件（一致性校验 / Story 拆分 / 编码上下文包），它们是同一阶段的三个产物，不计为独立阶段。
>
> 关键变化：**先数据/接口、再视觉、最后原型，最后挂营销**。原型不再是"先画一个看看"，而是把所有上游规范在静态页上的检验；E 阶段把已经成型的产品规范挂到 SEO + GEO 引擎上，确保上线即可被搜索引擎和 AI 引擎找到。

---

## 四、阶段角色

| 阶段 | 角色 | 单一职责 | 严禁做的事 |
|------|------|---------|-----------|
| R | 需求分析师 | 把模糊业务 → 结构化需求 + 流程图 | 不做技术选型、不画 UI |
| A | 架构师 | 选栈、定规范、定目录 | 不写功能字段、不写页面 |
| P | 安全/权限工程师 | 定角色、定认证、定授权 | 不实现具体业务接口 |
| D | 数据建模师 | 表结构、枚举、状态机、校验 | 不定接口、不定页面 |
| L | 接口设计师 | 接口契约、错误码、并发策略 | 不写 SQL、不写 UI |
| X | 体验总监 | 产品调性、参照、品牌情绪 | 不画具体组件 |
| S | UI 设计师 | Token、组件、布局、状态色 | 不定接口、不定页面交互 |
| I | 信息架构师 | 路由、页面清单、导航 | 不画单页布局 |
| N | 交互设计师 | 单页布局/元素/行为/4 态 | 不定接口、不写代码 |
| H | 原型工 | 出可点通 HTML | 不动业务规范 |
| V | QA + Scrum Master | 跨文件校验、拆 Story、打编码包 | 不写业务代码 |
| E | 增长 / SEO + GEO 工程师 | 出 SEO + GEO 营销执行包（关键词、技术 SEO、Schema、llms.txt、AI 引擎、UGC、90 天计划） | 不动产品功能、不写业务代码 |

> 一个人扮演多个角色没问题，关键是**开新对话窗口**时角色单一。

---

## 五、阶段闸门（Gate）

每阶段必经闸门才能进入下一阶段。任何 Gate 不通过 → 回到本阶段重做，**严禁强行往下推**。

| Gate | 通过判定 |
|------|---------|
| G-R | 需求基线无 `[待确认]`、覆盖主路径与异常路径、用户签字 |
| G-A | 架构规范无遗留问题、数据库/API/编码规范全部锁死 |
| G-P | 角色枚举与认证策略锁死，与 R 阶段角色清单一致 |
| G-D | 单功能数据规范字段齐全、状态机闭合、校验完整 |
| G-L | 单功能接口规范覆盖所有操作、错误码完备、与 D 字段一致 |
| G-X | 体验定调有 ≥3 个参照、有"反例"、有可验证的形容词 |
| G-S | 设计 Token 完备、所有组件都有 4 态、与 X 调性吻合 |
| G-I | 路由覆盖所有需求、每条需求都被某页承载 |
| G-N | 单页面规范元素齐全、4 态齐全、引用 D/L/S/P 全部对齐 |
| G-H | 原型可点通、4 态齐全、用户三轮内不再大改 |
| G-V | 一致性报告无红色项、Story ≤ 半天、编码包 ≤ 1200 行 |
| G-E | 关键词矩阵覆盖 P0 市场、技术 SEO 检查表全绿、AI 引擎被引列表 ≥ 5 个平台、有 90 天 Sprint 计划与 KPI |

---

## 六、最小用法

```
建好 docs/ 目录 → 复制 S01-R01 填需求 → AI 出 S01-R02 提问 → 你答 → AI 出 S01-R03 基线 →
依次跑 A/P/D/L/X/S/I/N → H 出原型 → V 出校验 → 进入编码 →
上线前/上线时跑 E：复制 S12-E01 填营销目标 → AI 出 S12-E02 提问 → 你答 → AI 出 S12-E03 SEO+GEO 方案 →
按 90 天计划执行
```

每一步只做一件事，AI 只看当下需要的文件。
