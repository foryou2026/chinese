# D01-01 AI 输出：数据规范

> **阶段**：D01·D 数据建模
> **上游**：B01(DB规范) + 各系统 C01/C02/C03
> **落盘**：共用 `docs/D01-data/common/` + 按系统 `docs/D01-data/<system-id>/`（均为多文件目录）

---

## 触发提示词

```
扮演"数据建模师"。
上游（已冻结）：各系统 C01 需求、C02 交互、C03 原型、B01 DB 规范。
模块：<module-id>，功能：<feature-id>，关联 R-ID：<...>
按 /prompt/D-develop/D01-01-AI输出-数据规范.md 输出。

第一步：识别共用数据（表/实体跨系统共享，各系统读写同一份数据）
      → 落盘 docs/D01-data/common/<module-id>/<feature-id>/
第二步：按系统产出系统专属数据
      → 落盘 docs/D01-data/<system-id>/<module-id>/<feature-id>/
```

---

## 共用 vs 系统专属的判定原则

| 类型 | 判定条件 | 落盘目录 | 典型示例 |
|------|---------|---------|---------|
| 共用 | 表/实体跨系统共享，各系统读写同一份数据，字段定义完全一致 | `docs/D01-data/common/` | auth.users 扩展表、全局配置表、共享枚举 |
| 系统专属 | 仅特定系统使用，或同一业务在不同系统有独立字段/约束/索引 | `docs/D01-data/<system-id>/` | app 端学习记录表、admin 端操作日志表 |

> 共用表在数据库中只存在一份，各系统通过 RLS 或接口层控制访问范围。
> 系统专属的 00-index.md 中引用共用实体清单，自检时需合并覆盖。

---

## AI 行为约束

1. 状态枚举来自 C02，只建枚举列+DB 约束，不重定义状态机
2. 不写路由/接口/HTML
3. 命名遵守 B01
4. 不重定义 B01 全局表，仅引用
5. 枚举与 C02 SM 不一致时必须声明
6. 未决项写 §99
7. **精简原则**：字段说明≤10 字、校验规则一句封顶、空节写"无"、禁止重复上游已有信息
8. 迁移文件落盘于 `system/` 下对应后端工程目录，禁止放在项目根目录
9. **生成后强制自检**：完成全部输出后，必须逐条对照 C01 需求清单（每条 R-ID）和 C02 状态机（每条转移），确认每条 R-ID 均有实体或规则承接、枚举值与 SM 一致。发现不一致或遗漏必须在 §99 声明
10. **共用优先**：先识别共用数据实体，再产出系统专属实体，避免跨系统重复定义

## 输出结构（多文件）

### 目录结构

```
docs/D01-data/
├── common/<module-id>/<feature-id>/     # 共用数据实体
│   ├── 00-index.md                      # 共用总览：ER图 + 实体清单 + 适用系统
│   ├── 01-<entity-name>.md              # 实体定义（每张表一个文件）
│   ├── 02-<entity-name>.md
│   ├── ...
│   └── enums.md                         # 共用枚举定义
├── app/<module-id>/<feature-id>/        # app 系统专属
│   ├── 00-index.md                      # 专属总览：ER图 + 引用共用 + 自检
│   ├── 01-<entity-name>.md
│   ├── ...
│   └── enums.md                         # app 专属枚举
└── admin/<module-id>/<feature-id>/      # admin 系统专属
    ├── 00-index.md
    ├── 01-<entity-name>.md
    ├── ...
    └── enums.md                         # admin 专属枚举
```

> 实体文件按业务重要性排序编号。每文件独立可审阅。
> 若某系统无专属数据实体，该系统目录可不创建。
