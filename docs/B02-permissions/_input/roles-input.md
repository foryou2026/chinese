# 角色输入（B02-P01）

> 阶段：B02-P 权限基线 · 用户输入  
> 确认 by PM · 2026-04-28

---

## 1. 角色清单（按 PM 决策最终拍板）

**只要 2 个角色，不要多角色矩阵**：

| 标识 | 名称 | 入口端 |
|------|------|-------|
| `super_admin` | 超级管理员 | 管理端（`web-admin` / `/admin/*`）|
| `user` | 用户 | 应用端（`web-app` / `/`）|

> 任何"运营 / 审核 / 客服 / 内容编辑"诉求，**本期一律视为超管兼任**。后续真的有人力分工诉求再开 G3 变更单。

## 2. 不要的东西

- ❌ 第三种角色（运营 / 审核 / 客服 / 内容 / AI 训练师 …全都不要）。
- ❌ RBAC 权限矩阵（菜单 × 操作 × 数据范围）。
- ❌ 按钮级权限配置（不要让管理员去后台开关按钮）。
- ❌ 2FA / TOTP（本期不上）。
- ❌ "我的设备"页（有问题重新登录即可）。
- ❌ Token 黑名单。

## 3. 关键约束

- **管理端的"用户管理"列表**：只能看 `role='user'` 的人，**绝对不能列出任何 super_admin**（包括自己）。SQL 上必须强约束，不能只靠前端过滤。
- **超管不能删除自己**；**超管不能禁用自己**。但可以删除/禁用其他 super_admin（多超管逆生场景）。
- **超管账号 seed**：账号 + 初始密码放 `.env`，首次部署 seed 写入；忘密 / 丢账 → 运维手工 SQL 重新插入。
- **凭证存储**：必须 `HttpOnly Cookie`（安全优先于便利性，避免 XSS 偷 token）。
- **多设备**：硬编码 3 个会话上限，第 4 次登录踢最早；不做"我的设备"页，让用户重新登录即可。
- **被禁用账号**：禁用动作 = `is_disabled=true` + 撚销全部 refresh + 删 user_sessions + 写审计。
- **i18n 全 5 语**：所有 `AUTH_*` 错误码与邮件模板 zh / en / vi / th / id 全量。

## 4. 登录方式

| 端 | 方式 |
|----|------|
| 应用端 | 邮箱密码 + Google OAuth |
| 管理端 | **仅邮箱密码**（不暴露 Google）|

## 5. 防爆破

- 本期**不接**任何第三方验证码（Cloudflare Turnstile / hCaptcha 一概不上）。
- 仅靠后端节流：
  - 登录失败 5 次 / 15 min（按 `email + ip`）→ 锁定到窗口结束；
  - 忘记密码：同邮箱 60s ≤ 1 次，同 IP 1h ≤ 5 次；
  - 注册：同 IP 1h ≤ 5 次。
