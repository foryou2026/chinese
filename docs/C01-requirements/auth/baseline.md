## 0. 摘要

- 给谁：知语应用端（18-35 岁东南亚成年用户）与运营管理端（内部管理员）。
- 做什么：邮箱密码与 Google 一键的注册登录、会话治理（多设备上限）、密码自助找回 / 修改、路由守卫、个人资料维护。
- 最大价值：用最低门槛、最强守卫，把"能进得来 + 进得安全 + 跨端一致"的账户体验做成可信底座。
- 最大风险：双 surface 共用账号系统时角色 / 会话隔离与"管理员入口被普通用户误用"风险，需在路由守卫 + 会话 + 审计层共同兜住。

## 1. 产品定义与目标

### 1.1 一句话定义

知语的统一账户与会话能力：让用户在 app 端自助注册登入、让管理员在 admin 端凭邀请账号安全登入，并由会话与守卫保证两端互不串扰。

### 1.2 业务目标

| ID | 目标 | 度量口径 | 验证时点 |
|----|------|--------|---------|
| O1 | 新用户首次完成注册 + 邮箱验证 + 登入 | 注册→首次登入完成率 ≥ 70% | 首版上线后 4 周 |
| O2 | 管理员登录失败率可控 | 单日"非管理员尝试登录管理端" / "全部管理端登录" ≤ 1% | 首版上线后 4 周 |
| O3 | 会话稳定性 | 30 天内被动登出（会话异常 / 错踢）≤ 0.5% | 持续监控 |
| O4 | 密码自助找回闭环 | 用户进入"找回密码"后 24 小时内成功改新密码并登入 ≥ 80% | 首版上线后 4 周 |

### 1.3 用户类型

| 类型 ID | 名称 | 画像（≤2 行） | 核心诉求 |
|--------|------|-----------|---------|
| U1 | 普通学习者（app） | 18-35 岁东南亚成年人，移动优先，部分用 Google 帐户 | 最快方式注册 / 找回密码并安全使用账号 |
| U2 | 运营管理员（admin） | 内部团队成员，邀请制账号，桌面优先 | 稳定凭据登录、被严格守卫，但不被会话频繁打断 |

## 3. 业务场景（叙述）

### 3.1 主要场景

普通用户首次使用：在 app 登录页选择"邮箱注册"或"Google 一键登录"。邮箱注册需收信验证后才能进入；Google 注册回调成功即登入。登入成功进入下游业务。

管理员日常登入：在 admin 登录页凭邀请账号登入，系统校验"管理员"身份后进入工作台。普通用户若误用 admin 入口登入会被立即阻断并提示去用户入口。

自助找回密码：用户在登录页点击"忘记密码"，输入邮箱后收到一次性邮件链接（15 分钟内一次性有效），设置新密码后自动登入并踢出其它设备。

修改 / 维护账号：登录后可在个人中心修改密码 / 维护个人资料（仅 app 端含资料维护），可选择"本设备退出"或"全部设备退出"。

### 3.2 隐含的重点能力（供 C02-IN 拆解）

- 跨 surface 共用账号但独立会话计数
- 节流 + 自锁 + 禁用三层防滥用
- 守卫层在 app / admin 各自的"登录页 redirect"约定
- 错误码 i18n 与文案一致性（5 语）

## 5. 角色定义

### 5.1 角色枚举

全系统**仅 2 个角色**：`ROLE-ADMIN`（管理端）+ `ROLE-USER`（应用端）。

| 角色 ID | DB / JWT 字面量 | 显示名 | 职责（一句话） | 不能做 | 入口 surface |
|--------|---------------|-------|-------------|--------|-------------|
| ROLE-USER | `user` | 普通用户 | 注册 / 登录 / 找回密码 / 修改密码 / 维护个人资料 / 进入受守卫的 app 页面 | 进入 admin 受守卫页面 | `app`（`/`） |
| ROLE-ADMIN | `admin` | 超级管理员 | 登录 / 找回密码 / 修改密码 / 退出 / 访问全部 admin 受守卫页面 | 在管理端自助注册；从管理端编辑普通用户密码 | `admin`（`/admin/*`） |

> **当前严禁引入第三种角色**。运营 / 审核 / 客服 / 内容编辑等诉求一律视为 admin 兼任。

### 5.2 多角色策略

- 一个用户拥有**唯一**角色（`user` 或 `admin`），不支持多角色叠加。
- 角色字段位置：JWT 元数据 `.role`；DB `users.role`；触发器同步到用户档案（冗余索引）。

### 5.3 角色 × surface 矩阵

| 角色 ID | `app` | `admin` |
|---------|-------|---------|
| ROLE-USER | ✅ 主 surface | ❌ 被守卫阻断 |
| ROLE-ADMIN | ❌ 不适用 | ✅ 主 surface |

### 5.4 超管保护规则

- 不能删 / 禁用自己（`AUTH_SUPER_ADMIN_SELF_DELETE`）。
- 不出现在任何"用户管理"列表中。
- UI 无创建管理员账号的入口；管理员仅可由运维 seed 流程创建。

## 6. 权限矩阵

### 6.1 全局权限矩阵（页面 / 功能点级）

| 资源 / 操作 | ROLE-USER | ROLE-ADMIN |
|------------|-----------|-----------|
| app 公开页（落地页 / 营销）| ✅ | ✅ |
| app 受守卫页（课程 / 个人中心等）| ✅（已登录）| ❌（入口不同）|
| app 个人资料 | ✅（仅自己）| ❌ |
| admin 受守卫页（全部）| ❌ | ✅（已登录）|
| 注册账号 | ✅（邮箱 / Google）| ❌（仅运维 seed）|
| 登录 | ✅（app 入口）| ✅（admin 入口）|
| 找回密码 | ✅ | ✅ |
| 修改密码 | ✅（已登录）| ✅（已登录）|
| 退出登录 | ✅ | ✅ |

### 6.2 数据可见范围原则

| 角色 ID | 默认可见范围 | 说明 |
|---------|-----------|------|
| ROLE-USER | owner-only | 只看自己的会话 / 资料 |
| ROLE-ADMIN | global（排除其他管理员）| 全平台 `role='user'` 用户；绝不列出 admin |

### 6.3 授权校验机制

> Token / 会话认证基础设施引用 `B01-architecture/09-auth-infra.md`，本节只定义授权层。

**后端（4 个中间件）**：

| 中间件 | 作用 | 挂在哪 |
|--------|------|--------|
| `authRequired` | 验证 JWT 有效并注入 `req.user` | app 受保护路由 |
| `adminRequired` | 在 `authRequired` 基础上校验 `role === 'admin'` | admin 全部路由 |
| `csrfRequired` | 验证 CSRF 令牌 | 所有写操作 |
| `optionalAuth` | 尝试解析 JWT，失败不报错 | 公开但需识别身份的路由 |

**前端路由守卫（TanStack Router）**：

- `/_authed/`：未登录 → 跳 `/auth/login?redirect=...`
- `/_admin/`：未登录 → 跳 `/admin/auth/login`；已登录但非 admin → 跳 `/admin/auth/login?reason=not_admin`

**按钮 / 操作级**：无权按钮直接不渲染（不灰显）；后端必须二次校验，前端不可信。

### 6.4 用户管理硬约束

- 后端 SQL 必须强约束 `WHERE role = 'user'`（不允许靠前端过滤代替）。
- 接口 / 列表 / 封禁 / 重置密码操作**只对 `role='user'` 生效**。
- **反例（禁止）**：返回全表给前端，让前端 `.filter(role !== 'admin')`。

## 8. 非功能性需求（NFR）

| 类型 | 指标 | 验收口径 |
|------|------|---------|
