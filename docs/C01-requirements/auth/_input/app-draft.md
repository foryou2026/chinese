# `auth` · R01 用户输入

> 确认 by PM · 2026-05-16

---

## 1. 产品想解决的问题

让"对中国语言文化感兴趣的东南亚用户"在 1 分钟内**拥有一个账号**，并随时**安全地**回来继续学习；同时给超管一条独立、低风险的登录通道。

## 2. 期望的用户旅程（草稿）

1. 用户在落地页点「开始学」 → 跳 `/auth/register`；
2. 选「邮箱 + 密码」**或**「Google 一键」二选一；
3. 邮箱注册 → 收到验证邮件 → 点链接回站 → 自动登入并进入 onboarding；
4. Google → 不弹验证邮件，直接进入 onboarding；
5. 已登录用户关闭浏览器 / 30 天内回访 → 仍是登录态（refresh 在）；
6. 用户忘密码 → 在登录页点「忘记密码」→ 输邮箱 → 收到链接 → 设新密码 → 自动登入；
7. 用户进入「个人中心」可以改头像 / 显示名 / 偏好语言 / 改密码 / 退出登录。

## 3. 硬约束（PM 红线）

- **本期不上手机号 / 第三方账号合并 / 2FA / 设备管理页 / 第三方验证码**；
- **不支持 self-delete**（删账号需走客服工单）；
- **不支持改邮箱**（改邮箱走客服工单）；
- 邮件本期 mock，不接真实 SMTP；
- 多设备活跃会话硬上限 = 3。

## 4. 关心的失败场景

- 邮箱已注册 → 必须清晰提示「已注册，请去登录或找回密码」；
- 弱密码 → 注册按钮禁用 + 内联提示规则；
- 5 次密码错 → 锁 15 分钟，提示明确（不暴露是否邮箱存在）；
- 同时第 4 台设备登录 → 最早设备被踢，提示「您的账号在其他设备登录，已被退出」；
- 账号被禁用 → 登录页 Toast 给出原因 + 客服入口；
- OAuth 失败 / 取消 → 静默回登录页，不打扰；
- 验证邮件过期 / 重置链接过期 → 单独的「链接已过期」页 + 重新发起入口。

## 5. 想要的语气（沿用 B03）

- 现代 + 克制：不夸张、不"嗨翻"；
- 不羞辱用户：错密码不写"密码错啦~"；
- 给出下一步：错了告诉怎么改，等了给倒计时。

## 6. 与其它 feature 的边界

- onboarding（选学习语言 / HSK 起点）→ **不**属于本 feature，归 `user-account`（v2 再开）；
- 钱包 / 订单 / 邀请 / 学习进度 → 不属于本 feature；
- 管理端的注册 / 邀请 / 用户列表 / 启用禁用 → 不属于本 feature（归 `auth` + `admin-users`）。
