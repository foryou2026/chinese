# 04 · A00-04 · 文档目录规划（AI 输出落盘约定）

> 本框架所有 AI 产出物统一落到项目根目录 `/docs` 下。本文件给出完整目录树与每个文件的命名约定，AI 必须严格遵守。
>
> 命名总则：
> - B 阶段子目录带 `B0[1-3]-` 前缀（项目级，全局唯一）。
> - C 阶段（产品设计）子目录带 `C0[1-6]-` 前缀；除 `C02-permissions/` 为**全局合并**外，其余下分 `<feature>/`。
> - D 阶段（开发）子目录带 `D0[1-3]-` 前缀，下分 `<feature>/`（每个 feature 一份）。
> - A00-meta/ 用于一次性阶段与多 feature C 循环的澄清问题落盘。
> - 与 `/prompt/` 下 4 个文件夹（A-framework / B-foundation / C-product / D-develop）一一对应。

## 三、AI 落盘行为约束

每次 AI 输出最终文件时必须：

1. **声明落盘路径**：在文件头第一行写：
   ```
   <!-- TARGET-PATH: docs/D01-data/order-mgmt/01-tables.md -->
   ```
2. **多文件输出**：若一次产出多个文件，逐个用 `<!-- TARGET-PATH: ... -->` 标注；用户按标注切分保存。
3. **拆分原则**：单文件 ≤ 1200 行；超出按本规划的子目录结构拆。
4. **更新索引**：拆分后必须同时更新对应 `00-index.md`，列出所有子文件 + 一句话职责。
5. **更新全局索引**：C 阶段产物必须同步更新对应的全局索引（如 C05 更新 `_global-index.md`）；D02 阶段路由增量必须同步追加至 `docs/D02-api/_global-routes.md`。
6. **移交问题**：未决问题写入对应子目录的 `99-open-questions.md`，不是塞在正文。

## 四、变更与冻结

- **单文件冻结**：在文件头 `冻结状态` 改为 `已冻结`。
- **子目录冻结**：在该子目录 `00-index.md` 顶部写 `冻结状态: 已冻结 · YYYY-MM-DD · 签字: <name>`。
- **变更记录**：所有冻结/解冻写入 `docs/A00-meta/changelog.md`。
- **F 层变更广播**：B01~B03 任一文件变更，须在 `changelog.md` 列出受影响的所有已开工 feature，并触发它们的 D03 V 重跑。
- **C02 增量并入**：每个 feature 的 C02 产出都**增量合并**到 `docs/C02-permissions/` 三份核心文件（`01-roles.md` / `02-authz-mechanism.md` / `03-data-model.md`）；冲突时以顶部 「冻结」 状态为准，后来者须调整。

## 四.5、多端项目目录变体（multi-surface）

> 仅当 B01 `08-surfaces.md` 中 surface ≥ 2 时启用。单端项目跳过本节，按上面"一、根目录结构"原样使用。

### 触发条件
B01 `08-surfaces.md` 中 surface 数量 ≥ 2。

### 受影响的目录与变体形态

| 阶段 | 单端形态 | 多端形态 |
|------|---------|---------|
| C01 R | `C01-requirements/<feature>/baseline.md` | `C01-requirements/<feature>/baseline.md`（**仍单份**，业务意图整体）+ 可选 `<feature>/<surface>/notes.md`（端特定补充） |
| C02 P | `docs/C02-permissions/` 全局合并（feature 增量并入） | 同单端；但角色×surface 矩阵会在 `01-roles.md` 体现多端列 |
| C03 I | `C03-ia/<feature>/0X-*.md` | `C03-ia/<feature>/_shared/{state-machines.md, flows-shared.md}`（共享真相）+ `<feature>/<surface>/{01-feature-catalog.md, 02-flows.md, 04-pages.md, 05-navigation.md, 06-coverage-matrix.md}`（端特定）|
| C04 N | `C04-pages/<feature>/<page-id>.md` | `C04-pages/<feature>/<surface>/<page-id>.md` |
| C05 H | `C05-prototype/<feature>/...` | `C05-prototype/<feature>/<surface>/...`（每个 surface 独立 index.html / <page>.html；HTML 均相对路径引用 `docs/B03-design/prototype-style/`） |
| C06 E | `C06-prd/<feature>/...` | `C06-prd/<feature>/_shared/{glossary.md, business-rules.md}` + `<feature>/<surface>/{00-index.md, 01-overview.md, 06-page-specs/, ...}` |
| D01 D | `D01-data/<feature>/...` | `D01-data/<feature>/...`（**仍单份**，数据模型是单一真相）|
| D02 L | `D02-api/<feature>/...` | `D02-api/<feature>/<surface>/{02-overview.md, 03-endpoints/, 01-routes-delta.md}`，路由强制带前缀 `/api/<surface>/*` |
| D03 V | `D03-validation/<feature>/...` | `D03-validation/<feature>/<surface>/...`（每个 surface 独立 V01/V02/V03，但都引用同一 `_shared/` 与单份 R + D） |

### 共享与端特定的判定原则

- **强一致 → 共享层**（`_shared/`）：状态机、跨端业务规则、术语、数据模型、需求基线
- **端特定 → `<surface>/`**：导航、页面清单、单页交互、原型 UI、端 PRD overview、路由
- **跨端联动场景**：admin 改动触发 app 刷新等场景，**真相**位于 `_shared/state-machines.md`；admin 与 app 各自的 D02 路由调用同一份 D01 表

### feature 命名通用约定

所有 feature（业务功能、内容功能、登录/认证功能、设置功能……）一律平等对待，**没有任何一类被特殊命名**。命名规则只有一条：

- **feature-id**：小写 kebab-case，体现业务语义（如 `course`、`discover-china`、`auth`、`payment`、`settings`）。
- **多端共用的 feature**：按上表走多端形态，即 `<feature>/<surface>/` 拆子目录；单端不变。
- **如果某 feature 在某 surface 不出产物**：该 surface 子目录直接不存在即可，**不要** 单独造一个带 surface 前缀的新 feature。

> **不预设登录**：如果项目没有登录/会话/账户体系，则不存在 auth 类 feature，下游一切照常；B01 `09-auth-infra.md` 留 "N/A"，C02 运行时可仅定「隐式」角色（如 `anonymous` / `guest` / `editor`）。
