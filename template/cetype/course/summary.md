# 课程模块二级架构规划

> 原型文件: 每个L1知识类型一个文件夹，内含各L2学习形式的独立应用端+管理端原型

## 全量统计

| 维度 | 数量 |
|------|------|
| L1 知识类型 | 10种 (免费9 + 含付费功能1) |
| L2 学习形式 | **18种** (免费17 + 付费1) |
| 原型页面总数 | 18 × 2面(应用端+管理端) = **36个原型面板** |

## 架构设计决策

### 为什么采用二级而非三级？

考试模块采用 **L1(题型) → L2(知识类型) → L3(变体)** 三级架构，因为14种题型 × 6种知识类型产生了大量的交叉组合(79种L3)。

课程模块的核心逻辑不同：用户按**知识类型**学习，而非按**学习形式**浏览。每种知识类型下的学习形式数量有限(1-3种)，不存在大规模交叉组合，因此二级架构足够：

- **L1 知识类型** (学什么) — 10种：拼音、偏旁部首、笔画笔顺、汉字、词语、成语、短句、语法句型、对话场景、阅读理解
- **L2 学习形式** (怎么学) — 1-3种/L1：卡片展示、专项训练、交互练习等

### 与考试模块的对称关系

```
课程模块: 知识类型 → 学习形式  (教内容 → 用什么方式教)
考试模块: 题型 → 知识类型 → 变体  (考什么格式 → 考什么内容 → 具体怎么考)
```

同一个知识领域(如拼音)，在课程模块中是**学习对象**，在考试模块中是**考察对象**。两个模块共享知识领域，但交互逻辑完全不同。

| 对话/角色扮演 | 无真实对话 | 有对话+角色扮演 | 有对话练习 | 有对话 | 对话学习+AI角色扮演(付费) |

| **学习形式分级** | 单一(每题固定) | **有**(学习/练习/测试) | **有**(笔记/课/对话/口语) | **有**(学习/复习分离) | **有**(L2学习形式) |

**关键洞察**: 所有成熟竞品都将同一知识内容按不同学习形式拆分。HelloChinese的拼音课就分3个阶段，ChineseSkill每单元含4种活动类型。这验证了我们采用L2学习形式的架构决策。

## 核心战略: "一次生成，无限分发"

**18种L2学习形式中，17种完全不需要实时AI调用。** 所有内容均可在后台预生成后静态分发:

| 数据类型 | 生成方式 | 分发策略 |
|---------|---------|---------|
| 文本数据(拼音/汉字/词语/成语/语法) | 结构化数据批量导入(CSV/JSON) | CDN静态分发 |
| 音频数据 | TTS批量生成 或 人工录制 | CDN音频文件 |
| 笔画/动画数据 | SVG/Lottie预渲染(Make Me a Hanzi等开源库) | 前端播放 |
| 描红判定 | Canvas + DTW算法 | **纯前端计算** |
| 分词/标注 | 后台NLP预处理 | 结构化JSON |

**唯一需要实时AI的场景**: 对话场景中的「角色扮演」语音评分(STT转录 + 发音评分)，归类为高级付费用户功能。

## 管理端核心配置模式

所有知识类型遵循统一配置范式:

1. **基础字段**: 内容本体 + 拼音 + 中英释义
2. **媒体资源**: 音频上传/TTS + 图片/动画(SVG/Lottie)
3. **分类标签**: HSK等级 + 主题 + 难度 + 排序
4. **关联关系**: 知识点之间的引用关联(如汉字↔部首, 词语↔汉字)
5. **批量操作**: CSV/JSON导入 + AI辅助批量生成
